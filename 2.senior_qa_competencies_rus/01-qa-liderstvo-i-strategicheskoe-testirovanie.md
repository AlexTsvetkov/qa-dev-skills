## 1. Тест-стратегия и тест-план

### Тест-стратегия vs Тест-план

| Аспект | Тест-стратегия | Тест-план |
|--------|---------------|-----------|
| **Область** | Организация/проект | Конкретный релиз/спринт |
| **Уровень** | Высокоуровневый подход | Детальное выполнение |
| **Автор** | QA Lead / Manager | QA Lead / Senior QA |
| **Содержание** | Виды тестирования, инструменты, среды | Расписание, ресурсы, конкретные тесты |
| **Частота обновления** | Квартально/по проекту | Каждый спринт/релиз |

### Шаблон тест-стратегии:

```markdown
# Тест-стратегия — [Название проекта]

## 1. Обзор
- Цели проекта и определение качества
- Заинтересованные стороны и их ожидания

## 2. Объём тестирования
- Что включено / исключено
- Уровни тестирования (модульное, интеграционное, E2E)
- Виды тестирования (функциональное, производительность, безопасность)

## 3. Подход к автоматизации
- Что автоматизируем vs ручное тестирование
- Фреймворки и инструменты
- Стратегия поддержки

## 4. Среды и данные
- Тестовые среды (Dev, Staging, Pre-prod)
- Стратегия управления тестовыми данными

## 5. Критерии входа/выхода
- Когда начинается тестирование
- Когда тестирование завершено

## 6. Управление рисками
- Выявленные риски и план митигации

## 7. Метрики и отчётность
- KPI для отслеживания
- Частота отчётности
```

## 2. Матрица трассируемости

### Зачем нужна трассируемость

Матрица трассируемости требований (RTM) связывает требования с тест-кейсами, обеспечивая полное покрытие.

```
Требование → Тест-кейс → Результат выполнения → Дефект

REQ-001: Вход по email/паролю
  ├── TC-001: Успешный вход ✅
  ├── TC-002: Неверный пароль ✅
  ├── TC-003: Блокировка аккаунта ❌ → BUG-042
  └── TC-004: Сброс пароля ✅

REQ-002: Оплата кредитной картой
  ├── TC-010: Успешная оплата Visa ✅
  ├── TC-011: Отклонённая карта ✅
  └── TC-012: Истёкший срок карты ⏳ (не выполнен)
```

### Метрики покрытия:

| Метрика | Формула | Цель |
|---------|---------|------|
| Покрытие требований | Требования с тестами / Всего требований | > 95% |
| Покрытие выполнения | Выполненные тесты / Всего тестов | > 95% |
| Процент успешных | Успешные тесты / Выполненные тесты | > 90% |
| Покрытие дефектами | Требования с дефектами / Всего требований | Мониторинг тренда |

## 3. Планирование, оценка и приоритизация

### Техники оценки тестирования:

| Техника | Описание | Когда использовать |
|---------|----------|-------------------|
| **По прецедентам** | На основе прошлых похожих проектов | Повторяющиеся проекты |
| **По тест-кейсам** | Часы × количество тест-кейсов | Определённые требования |
| **По процентам** | Тестирование = 25-35% от разработки | Раннее планирование |
| **Трёхточечная** | (Оптим. + 4×Реальн. + Пессим.) / 6 | Неопределённый объём |

### Приоритизация:

```
Матрица приоритетов:
                    Высокое влияние    Низкое влияние
Высокий риск    │  P1: Тестировать   │  P2: Тестировать   │
                │  первым и глубоко  │  тщательно          │
                ├────────────────────┼─────────────────────┤
Низкий риск     │  P3: Базовое       │  P4: Смоук-тест     │
                │  покрытие           │  или пропустить     │
```

## 4. Метрики качества и KPI

### Ключевые метрики:

| Метрика | Формула | Хорошо | Плохо |
|---------|---------|--------|-------|
| **Показатель утечки** | Дефекты в проде / Все дефекты | < 5% | > 15% |
| **Плотность дефектов** | Дефекты / Тысяча строк кода | < 5 | > 20 |
| **Процент автоматизации** | Авто-тесты / Все тесты | > 60% | < 20% |
| **Уровень нестабильности** | Нестабильные / Все авто-тесты | < 2% | > 10% |
| **Время до фидбека** | Время от коммита до результатов | < 15 мин | > 2 часов |

## 5. Тестирование на основе рисков

### Матрица оценки рисков:

| Фактор | Высокий риск | Низкий риск |
|--------|-------------|-------------|
| Сложность кода | Сложная бизнес-логика | Простой CRUD |
| Частота изменений | Часто модифицируется | Стабильный код |
| История дефектов | Частые баги | Редкие баги |
| Влияние на пользователя | Оплата, аутентификация | Страница «О нас» |
| Интеграции | Внешние API/сервисы | Изолированный модуль |

### Критерии выхода:

```markdown
## Критерии выхода для релиза

### Обязательные (блокирующие):
- [ ] Все критические/блокирующие дефекты закрыты
- [ ] Смоук-тесты: 100% успешно
- [ ] Регрессия: > 95% успешно
- [ ] Не найдены критические уязвимости

### Желательные:
- [ ] Покрытие тестов > 90%
- [ ] Нестабильность < 3%
- [ ] Производительность в пределах SLA

### Осведомительные (информирование о рисках):
- [ ] Список известных проблем задокументирован
- [ ] Мониторинг настроен
- [ ] План отката готов
```

## 6. QA-лидерство и менторство

### Обязанности QA-лидера:

| Область | Деятельность |
|---------|-------------|
| **Стратегия** | Определение подхода к тестированию проекта |
| **Менторство** | Развитие навыков junior QA-инженеров |
| **Процесс** | Улучшение QA-процессов и практик |
| **Коммуникация** | Отчёты о статусе качества стейкхолдерам |
| **Принятие решений** | Go/No-Go рекомендации по релизу |

### Фреймворк менторства:

```
1. ПОКАЖИ — Продемонстрируй технику
2. ОБЪЯСНИ — Почему этот подход работает и когда его применять
3. ПРАКТИКУЙ — Пусть попробуют под наблюдением
4. ОБРАТНАЯ СВЯЗЬ — Конкретная конструктивная обратная связь
5. САМОСТОЯТЕЛЬНОСТЬ — Постепенно снижай контроль
```

## 7. Коммуникация рисков качества

### Шаблон коммуникации рисков:

```markdown
## Риск: Надёжность обработки платежей
**Серьёзность**: Высокая | **Вероятность**: Средняя

### Текущее состояние
- Таймаут платежей: 3.2% (было 0.5%)
- Затронуто: оплата кредитной картой через Stripe

### Влияние на бизнес
- ~$15K/день потерянных транзакций
- 120 обращений в поддержку за неделю

### Варианты решения
1. Увеличить таймаут (1 день) — снижает на ~60%
2. Добавить логику повторов (3 дня) — снижает на ~90%
3. Переключить на резервный шлюз (1 неделя) — устраняет

### Рекомендация
Вариант 1 немедленно, затем Вариант 2 в следующем спринте.
```

## ✅ Вопросы для самопроверки

- [ ] В чём разница между тест-стратегией и тест-планом?

<details>
<summary>Ответ</summary>

**Тест-стратегия** — высокоуровневый документ, описывающий подход к тестированию на уровне организации/проекта. Определяет виды тестирования, инструменты, среды. Обновляется редко. **Тест-план** — детальный документ для конкретного релиза/спринта. Содержит расписание, ресурсы, конкретные тест-кейсы. Обновляется каждый спринт.

</details>

- [ ] Что такое тестирование на основе рисков и зачем оно нужно?

<details>
<summary>Ответ</summary>

Тестирование на основе рисков — подход, где объём тестирования пропорционален уровню риска. Риск оценивается по двум осям: **вероятность сбоя** и **влияние сбоя**. Области с высоким риском получают глубокое тестирование, области с низким — базовое покрытие. Это позволяет оптимально использовать ограниченное время и ресурсы тестирования.

</details>

- [ ] Какие метрики качества наиболее важны для senior QA?

<details>
<summary>Ответ</summary>

Ключевые метрики: (1) **Показатель утечки дефектов** — баги, ушедшие в прод (цель <5%). (2) **Покрытие тестами** по областям риска. (3) **Процент автоматизации** (цель >60%). (4) **Время до фидбека** — от коммита до результатов тестов. (5) **Тренды дефектов** — улучшается ли качество со временем. Метрики нужно представлять на дашборде и использовать для принятия решений.

</details>

- [ ] Как сообщать о рисках качества стейкхолдерам?

<details>
<summary>Ответ</summary>

Структура: (1) **Чётко обозначить риск** — что может пойти не так. (2) **Количественно оценить влияние** — в деньгах, пользователях, обращениях в поддержку. (3) **Представить варианты** — с оценкой усилий и трейд-оффами. (4) **Дать рекомендацию** — своё экспертное мнение. (5) **Поддержать решение** — какое бы ни было принято. Никогда не говорите «нельзя релизить» без данных. Адаптируйте уровень деталей к аудитории.

</details>

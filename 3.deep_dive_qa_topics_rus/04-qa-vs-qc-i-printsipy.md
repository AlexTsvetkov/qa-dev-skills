## 1. QA vs QC

```
  ┌─────────────────────────┬─────────────────────────┐
  │     QA (Assurance)      │     QC (Control)        │
  │     ОБЕСПЕЧЕНИЕ         │     КОНТРОЛЬ            │
  ├─────────────────────────┼─────────────────────────┤
  │ Фокус на ПРОЦЕССЕ       │ Фокус на ПРОДУКТЕ      │
  │ Превентивный подход     │ Реактивный подход       │
  │ "Делаем правильно"     │ "Сделали правильно?"   │
  │ Процессы, стандарты     │ Тестирование, проверки  │
  │ На протяжении всего     │ На этапе тестирования   │
  │ жизненного цикла        │                         │
  ├─────────────────────────┼─────────────────────────┤
  │ Примеры:                │ Примеры:                │
  │ • Определение процессов │ • Выполнение тест-кейсов│
  │ • Код-ревью             │ • Баг-репорты           │
  │ • Стандарты кодирования │ • Smoke-тестирование    │
  │ • Тренинги              │ • Регрессия             │
  │ • Аудиты процессов      │ • Приёмочное тестир.    │
  └─────────────────────────┴─────────────────────────┘

  Senior QA = сочетает ОБА подхода:
    QA: Улучшает процессы, внедряет стандарты
    QC: Выполняет и руководит тестированием
```

## 2. Семь принципов тестирования (ISTQB)

```
  1. Тестирование показывает наличие дефектов
     ┌──────────────────────────────────────────┐
     │ Тестирование может показать, что дефекты │
     │ ЕСТЬ, но НЕ может доказать, что их НЕТ  │
     └──────────────────────────────────────────┘

  2. Исчерпывающее тестирование невозможно
     ┌──────────────────────────────────────────┐
     │ Нельзя протестировать ВСЕ комбинации.    │
     │ Используйте анализ рисков и приоритеты.  │
     └──────────────────────────────────────────┘

  3. Раннее тестирование (Shift-Left)
     ┌──────────────────────────────────────────┐
     │ Начинайте тестирование как можно раньше. │
     │ Чем раньше найден баг = тем дешевле.     │
     └──────────────────────────────────────────┘

  4. Кластеризация дефектов
     ┌──────────────────────────────────────────┐
     │ 80% багов — в 20% модулей (Парето).      │
     │ Фокусируйтесь на проблемных областях.   │
     └──────────────────────────────────────────┘

  5. Парадокс пестицида
     ┌──────────────────────────────────────────┐
     │ Одни и те же тесты перестают находить     │
     │ новые баги. Регулярно обновляйте тесты.  │
     └──────────────────────────────────────────┘

  6. Тестирование зависит от контекста
     ┌──────────────────────────────────────────┐
     │ Подход зависит от типа проекта:          │
     │ Банк ≠ Игра ≠ Стартап ≠ Медицина        │
     └──────────────────────────────────────────┘

  7. Заблуждение об отсутствии ошибок
     ┌──────────────────────────────────────────┐
     │ Нет багов ≠ хороший продукт.             │
     │ Если система не нужна — баги не важны.   │
     └──────────────────────────────────────────┘
```

## 3. Тестовая пирамида

```
                    ╱╲
                   ╱  ╲         E2E / UI тесты
                  ╱ E2E╲        • Медленные, дорогие
                 ╱──────╲       • 10% от всех тестов
                ╱        ╲
               ╱ Интеграц.╲    Интеграционные тесты
              ╱────────────╲   • API, сервисы
             ╱              ╲  • 20-30% от всех тестов
            ╱   Юнит-тесты   ╲ Юнит-тесты
           ╱──────────────────╲ • Быстрые, дешёвые
          ╱                    ╲• 60-70% от всех тестов

  Принцип: Больше тестов внизу, меньше вверху

  ┌──────────────┬──────────┬───────────┬───────────┐
  │ Уровень      │ Скорость │ Стоимость │ Доля      │
  ├──────────────┼──────────┼───────────┼───────────┤
  │ Юнит         │ Быстро   │ Низкая    │ 60-70%    │
  │ Интеграция   │ Средне   │ Средняя   │ 20-30%    │
  │ E2E/UI       │ Медленно │ Высокая   │ 5-10%     │
  └──────────────┴──────────┴───────────┴───────────┘

  Анти-паттерн: "Перевёрнутая пирамида"
    Много E2E тестов, мало юнит-тестов
    → Медленная обратная связь
    → Нестабильные тесты (flaky)
    → Дорогая поддержка
```

## 4. Улучшение процессов

```
  Цикл непрерывного улучшения (PDCA):

  ┌──────────┐    ┌──────────┐
  │  Plan    │───→│   Do     │
  │Планируй  │    │Выполняй  │
  └──────────┘    └──────────┘
       ↑               │
       │               ▼
  ┌──────────┐    ┌──────────┐
  │  Act     │←───│  Check   │
  │Действуй  │    │Проверяй  │
  └──────────┘    └──────────┘

  Практики улучшения для QA:
    1. Ретроспективы после каждого спринта
       - Что работало хорошо?
       - Что можно улучшить?
       - Конкретные действия

    2. Анализ пропущенных багов (Escaped Defects)
       - Почему пропустили?
       - Как предотвратить?
       - Нужны ли новые тесты?

    3. Обзор тестовых метрик
       - Тренды улучшаются?
       - Где узкие места?
       - Какие процессы неэффективны?

    4. Обмен знаниями
       - Внутренние презентации
       - Документация
       - Парное тестирование
```

## 5. Soft skills для старшего QA

```
  Ключевые навыки:

  Коммуникация:
    ✓ Чёткая документация и артефакты тестирования
    ✓ Объяснение сложного простым языком
    ✓ Отчёты для разных аудиторий
    ✓ Конструктивная обратная связь

  Лидерство:
    ✓ Менторинг и обучение коллег
    ✓ Принятие решений под давлением
    ✓ Управление конфликтами
    ✓ Влияние без формальной власти

  Решение проблем:
    ✓ Анализ корневых причин (Root Cause Analysis)
    ✓ Критическое мышление
    ✓ Системное мышление
    ✓ Декомпозиция сложных проблем

  Работа со стейкхолдерами:
    ✓ Выравнивание ожиданий
    ✓ Управление приоритетами
    ✓ Эскалация рисков
    ✓ Go/No-Go рекомендации

  Пример упрощения для не-технической аудитории:
    Техническое: "Обнаружен race condition в модуле 
    оплаты при конкурентном доступе к БД"
    
    Бизнес: "Если два пользователя оплатят одновременно,
    один из платежей может потеряться. Мы это 
    исправляем и выпустим обновление завтра."
```

## ✅ Вопросы для самопроверки

1. В чём разница между QA и QC?

<details>
<summary>Ответ</summary>

**QA (Quality Assurance)** — фокус на процессе, превентивный подход: стандарты, процессы, код-ревью, тренинги. «Делаем правильно». **QC (Quality Control)** — фокус на продукте, реактивный подход: выполнение тестов, баг-репорты, регрессия. «Сделали правильно?». Senior QA сочетает оба подхода.

</details>

2. Перечислите 7 принципов тестирования ISTQB.

<details>
<summary>Ответ</summary>

(1) Тестирование показывает наличие дефектов, но не их отсутствие. (2) Исчерпывающее тестирование невозможно. (3) Раннее тестирование (Shift-Left). (4) Кластеризация дефектов (80/20 по Парето). (5) Парадокс пестицида — одни тесты перестают находить баги. (6) Тестирование зависит от контекста. (7) Заблуждение об отсутствии ошибок — нет багов ≠ хороший продукт.

</details>

3. Объясните тестовую пирамиду и почему «перевёрнутая пирамида» — анти-паттерн.

<details>
<summary>Ответ</summary>

**Тестовая пирамида:** основание — юнит-тесты (60-70%, быстрые, дешёвые), середина — интеграционные тесты (20-30%), вершина — E2E/UI тесты (5-10%, медленные, дорогие). **Перевёрнутая пирамида** — когда много E2E и мало юнит-тестов. Это анти-паттерн, потому что: медленная обратная связь, нестабильные тесты (flaky), дорогая поддержка.

</details>

4. Как проводить анализ пропущенных багов (Escaped Defects)?

<details>
<summary>Ответ</summary>

Анализируем каждый баг, дошедший до продакшена: (1) Почему пропустили — не было тест-кейса, недостаточное покрытие, плохие тестовые данные? (2) Как предотвратить — добавить тест-кейс, улучшить покрытие, обновить чеклист. (3) Нужны ли новые тесты — создать автотест или ручной кейс. Результаты обсуждаются на ретроспективе.

</details>

5. Как объяснить технический баг не-технической аудитории?

<details>
<summary>Ответ</summary>

Переводим техническое описание в бизнес-влияние. Вместо «race condition в модуле оплаты при конкурентном доступе к БД» говорим: «Если два пользователя оплатят одновременно, один платёж может потеряться. Мы исправляем это и выпустим обновление завтра». Фокус: что произойдёт для пользователя, какое влияние на бизнес, когда будет исправлено.

</details>

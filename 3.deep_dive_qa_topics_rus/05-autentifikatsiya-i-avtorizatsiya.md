## 1. Аутентификация vs Авторизация

```
  ┌──────────────────────────┬──────────────────────────┐
  │    АУТЕНТИФИКАЦИЯ        │    АВТОРИЗАЦИЯ           │
  │    (Authentication)      │    (Authorization)       │
  ├──────────────────────────┼──────────────────────────┤
  │ "КТО ты?"               │ "ЧТО тебе можно?"       │
  │                          │                          │
  │ Проверка личности        │ Проверка прав доступа    │
  │ Логин + Пароль           │ Роли и разрешения        │
  │ Выполняется ПЕРВОЙ       │ Выполняется ВТОРОЙ      │
  └──────────────────────────┴──────────────────────────┘

  Аналогия:
    Аутентификация = Показать паспорт на входе (кто ты?)
    Авторизация    = Проверить пропуск в здании (куда можешь?)

  Поток:
    Пользователь → Аутентификация → Авторизация → Доступ
         │              │                │            │
      Логин/Пароль   Проверка        Проверка     Ресурс
                     личности        прав
```

## 2. Методы аутентификации

```
  1. Basic Authentication:
     ┌──────────────────────────────────────────┐
     │ Authorization: Basic base64(user:pass)   │
     │ Просто, но НЕ безопасно без HTTPS        │
     │ Пароль отправляется с каждым запросом     │
     └──────────────────────────────────────────┘

  2. Token-Based (Bearer Token / JWT):
     ┌──────────────────────────────────────────┐
     │ 1. Логин → Получаем токен                │
     │ 2. Authorization: Bearer <token>         │
     │ 3. Токен имеет срок действия (expiry)    │
     │                                          │
     │ JWT = Header.Payload.Signature            │
     │ Header:  {"alg":"HS256","typ":"JWT"}     │
     │ Payload: {"sub":"user1","role":"admin"}  │
     │ Signature: HMACSHA256(header+payload)    │
     └──────────────────────────────────────────┘

  3. OAuth 2.0:
     ┌──────────────────────────────────────────┐
     │ Пользователь → Auth Server → Приложение  │
     │                                          │
     │ Потоки (Grants):                         │
     │ • Authorization Code — веб-приложения    │
     │ • Client Credentials — сервис-к-сервису  │
     │ • Implicit — устаревший (SPA)            │
     │ • PKCE — мобильные и SPA                 │
     └──────────────────────────────────────────┘

  4. SSO (Single Sign-On):
     ┌──────────────────────────────────────────┐
     │ Один логин — доступ ко всем приложениям  │
     │ Протоколы: SAML, OpenID Connect          │
     │ Пример: Вход через Google/Microsoft      │
     └──────────────────────────────────────────┘
```

## 3. Тестирование безопасности (OWASP Top 10)

```
  Ключевые уязвимости для QA:

  1. Broken Access Control (Нарушение контроля доступа)
     Тест: Обычный пользователь → запрос к /admin
     Ожидание: 403 Forbidden
     
  2. Injection (Инъекции)
     Тест: Ввод ' OR '1'='1 в поле логина
     Ожидание: Ошибка валидации, НЕ доступ

  3. Broken Authentication (Нарушенная аутентификация)
     Тест: Brute-force с неверными паролями
     Ожидание: Блокировка после N попыток

  4. Security Misconfiguration (Ошибки конфигурации)
     Тест: Доступ к /debug, /swagger, /.env
     Ожидание: 404 или 403 в продакшене

  5. XSS (Cross-Site Scripting)
     Тест: Ввод <script>alert('XSS')</script>
     Ожидание: Экранирование, скрипт НЕ исполняется

  Чек-лист безопасности для QA:
    ✓ HTTPS используется (нет HTTP)
    ✓ Пароли не хранятся в открытом виде
    ✓ Токены имеют срок действия
    ✓ Чувствительные данные не в URL
    ✓ CORS настроен правильно
    ✓ Rate limiting включен
    ✓ Сообщения об ошибках не раскрывают детали
```

## 4. RBAC и тестирование прав доступа

```
  RBAC = Role-Based Access Control

  Пример матрицы прав:
  ┌────────────────┬────────┬──────────┬───────┐
  │ Действие       │ Admin  │ Manager  │ User  │
  ├────────────────┼────────┼──────────┼───────┤
  │ Просмотр       │   ✅   │    ✅    │  ✅   │
  │ Создание       │   ✅   │    ✅    │  ❌   │
  │ Редактирование │   ✅   │    ✅    │  ❌   │
  │ Удаление       │   ✅   │    ❌    │  ❌   │
  │ Управление     │   ✅   │    ❌    │  ❌   │
  │ пользователями │        │          │       │
  └────────────────┴────────┴──────────┴───────┘

  Стратегия тестирования:
    1. Позитивные тесты — каждая роль выполняет
       разрешённые действия
    2. Негативные тесты — каждая роль пытается
       выполнить ЗАПРЕЩЁННЫЕ действия
    3. Эскалация привилегий — User пытается 
       получить права Admin
    4. Горизонтальный доступ — User1 пытается
       получить данные User2
```

## 5. Практические сценарии

```
  Сценарий 1: Тестирование JWT токена
    1. Логин с валидными данными → получаем токен
    2. Запрос с токеном → 200 OK
    3. Запрос с просроченным токеном → 401
    4. Запрос с изменённым токеном → 401
    5. Запрос без токена → 401

  Сценарий 2: Тестирование OAuth 2.0
    1. Редирект на auth сервер → корректная страница
    2. Логин → редирект с authorization code
    3. Обмен code на access token → успех
    4. Запрос с access token → 200 OK
    5. Использование истёкшего refresh token → 401

  Сценарий 3: Тестирование Rate Limiting
    1. Отправка 5 неверных паролей → ошибка логина
    2. Отправка 6-го → блокировка аккаунта
    3. Ожидание N минут → разблокировка
    4. Валидный логин → успех

  Сценарий 4: Тестирование CORS
    1. Запрос с разрешённого домена → 200 OK
    2. Запрос с неразрешённого домена → заблокирован
    3. Проверка заголовков Access-Control-Allow-Origin
```

## ✅ Вопросы для самопроверки

1. В чём разница между аутентификацией и авторизацией?

<details>
<summary>Ответ</summary>

**Аутентификация** — проверка личности («Кто ты?»): логин, пароль, токен. Выполняется первой. **Авторизация** — проверка прав доступа («Что тебе можно?»): роли, разрешения. Выполняется после аутентификации. Аналогия: паспорт на входе (аутентификация) → пропуск в здании (авторизация).

</details>

2. Сравните Basic Auth, JWT и OAuth 2.0.

<details>
<summary>Ответ</summary>

**Basic Auth** — логин:пароль в base64 с каждым запросом, просто но небезопасно без HTTPS. **JWT (Bearer Token)** — одноразовый логин, затем токен с payload (роль, срок действия), stateless, самодостаточный. **OAuth 2.0** — делегированная авторизация через отдельный auth-сервер, поддерживает разные потоки (Authorization Code, Client Credentials, PKCE), используется для «войти через Google».

</details>

3. Как тестировать RBAC (контроль доступа на основе ролей)?

<details>
<summary>Ответ</summary>

(1) **Позитивные тесты** — каждая роль выполняет разрешённые действия. (2) **Негативные тесты** — роль пытается выполнить запрещённые действия (ожидаем 403). (3) **Вертикальная эскалация** — User пытается получить права Admin. (4) **Горизонтальный доступ** — User1 пытается получить данные User2. Создаётся матрица ролей × действий.

</details>

4. Назовите 5 уязвимостей OWASP Top 10 и как их тестировать.

<details>
<summary>Ответ</summary>

(1) **Broken Access Control** — запрос к /admin от обычного пользователя → ожидаем 403. (2) **Injection** — ввод `' OR '1'='1` → ожидаем ошибку валидации. (3) **Broken Authentication** — brute-force → ожидаем блокировку после N попыток. (4) **Security Misconfiguration** — доступ к /debug, /.env → ожидаем 404/403 на проде. (5) **XSS** — ввод `<script>alert('XSS')</script>` → ожидаем экранирование.

</details>

5. Опишите сценарий тестирования JWT токена.

<details>
<summary>Ответ</summary>

(1) Логин с валидными данными → получаем JWT токен. (2) Запрос с валидным токеном → 200 OK. (3) Запрос с просроченным токеном → 401 Unauthorized. (4) Запрос с изменённым/поддельным токеном → 401 Unauthorized. (5) Запрос без токена → 401 Unauthorized. Дополнительно: проверить payload на отсутствие чувствительных данных, проверить срок действия.

</details>

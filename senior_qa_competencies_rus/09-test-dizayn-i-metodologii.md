## 1. Техники тест-дизайна

### Анализ граничных значений (BVA):

```
Поле: Возраст (допустимо 18-65)

Граничные значения для тестирования:
  17 → невалидно (ниже границы)
  18 → валидно (нижняя граница)
  19 → валидно (рядом с границей)
  64 → валидно (рядом с границей)
  65 → валидно (верхняя граница)
  66 → невалидно (выше границы)

Также проверить: 0, -1, пустое, максимальное число
```

### Эквивалентное разбиение:

```
Поле: Скидка по возрасту

Классы эквивалентности:
  [0-17]   → Детская скидка 50%
  [18-64]  → Стандартная цена
  [65-120] → Пенсионная скидка 30%
  [<0]     → Невалидный ввод
  [>120]   → Невалидный ввод

Из каждого класса берём 1 значение:
  10, 35, 80, -5, 200
```

### Таблицы решений:

```
Условия логина:

| Условие              | Правило 1 | Правило 2 | Правило 3 | Правило 4 |
|----------------------|-----------|-----------|-----------|-----------|
| Верный логин?        | Да        | Да        | Нет       | Нет       |
| Верный пароль?       | Да        | Нет       | Да        | Нет       |
| Действие             | Вход      | Ошибка    | Ошибка    | Ошибка    |
```

### Диаграммы состояний и переходов:

```
Состояния заказа:

  [Создан] --оплата--> [Оплачен] --отправка--> [Отправлен]
     |                     |                        |
  --отмена-->           --возврат-->             --доставка-->
     |                     |                        |
  [Отменён]           [Возвращён]              [Доставлен]

Тестируем:
✅ Все допустимые переходы
✅ Все недопустимые переходы (Создан → Доставлен = ошибка)
✅ Конечные состояния (Отменён, Возвращён, Доставлен)
```

## 2. Исследовательское тестирование

### Сессионное тестирование (SBTM):

```
Хартия сессии:
  ИССЛЕДОВАТЬ: модуль оплаты
  ИСПОЛЬЗУЯ: различные типы карт и суммы
  ЧТОБЫ ОБНАРУЖИТЬ: проблемы с обработкой платежей

Длительность: 60 минут
Тестировщик: Настя

Заметки:
  0:00 — Начало. Тестирую Visa, Mastercard, Mir
  0:15 — Visa и MC работают. Mir → ошибка 500
  0:25 — Mir с суммой < 100₽ работает, > 100₽ — нет
  0:40 — Нашла: лимит Mir = 99₽, должен быть 999999₽

Результат:
  Баги: 1 критический (лимит Mir)
  Покрытие: 4 платёжные системы
  Области для доп. тестирования: валюты, 3D Secure
```

### Эвристики исследовательского тестирования:

| Эвристика | Что проверять |
|-----------|--------------|
| **Граничные значения** | 0, 1, max, max+1, пустое |
| **Неожиданный ввод** | Emoji, unicode, HTML-теги |
| **Прерывания** | Обновить страницу, закрыть вкладку |
| **Конкуренция** | Две вкладки, одновременные действия |
| **Конфигурация** | Разные браузеры, языки, часовые пояса |
| **CRUD** | Создать, прочитать, обновить, удалить |

## 3. Тестовая пирамида

### Уровни пирамиды:

```
         /  E2E  \          ← Мало, медленные, дорогие
        /----------\
       / Интеграция \       ← Средне
      /----------------\
     /   Модульные       \  ← Много, быстрые, дешёвые
    /____________________\

Модульные (70%):
  - Одна функция/метод
  - Миллисекунды
  - Пишут разработчики

Интеграционные (20%):
  - Взаимодействие компонентов
  - API-тесты, тесты БД
  - QA + разработчики

E2E (10%):
  - Полный путь пользователя
  - Реальный браузер
  - Пишут QA
```

### Антипаттерн — перевёрнутая пирамида:

```
     /____________________\   ← Много E2E — медленно, нестабильно
      \  Интеграция      /    ← Мало
       \________________/
        \  Модульные  /       ← Почти нет
         \__________/

Проблемы:
- Тесты выполняются часами
- Много нестабильных (flaky) тестов
- Трудно найти причину падения
- Дорогое обслуживание
```

## 4. Улучшение QA-процессов

### Цикл непрерывного улучшения:

```
1. ИЗМЕРЬ — Собери данные (время, покрытие, баги)
2. АНАЛИЗИРУЙ — Найди узкие места
3. ПРЕДЛОЖИ — Конкретные улучшения
4. ВНЕДРИ — Маленькими шагами
5. ПРОВЕРЬ — Измерь результат
6. ПОВТОРИ
```

### Примеры улучшений:

| Проблема | Метрика | Улучшение |
|----------|---------|-----------|
| Долгий CI | Тесты: 45 мин | Параллелизация → 12 мин |
| Много flaky | 15% нестабильных | Карантин + починка → 2% |
| Пропущенные баги | 5 багов в проде/мес | Ревью тест-кейсов → 1 баг |
| Поздние баги | Баги на регрессии | Shift-left → баги в спринте |

## ✅ Вопросы для самопроверки

- [ ] Объясните разницу между граничными значениями и эквивалентным разбиением.

<details>
<summary>Ответ</summary>

**Эквивалентное разбиение** делит входные данные на классы, где поведение одинаково, и берёт одно значение из каждого класса. **Граничные значения** проверяют значения на границах классов, где чаще всего ошибки (min, min-1, max, max+1). Они дополняют друг друга: разбиение определяет классы, BVA уточняет критические точки.

</details>

- [ ] Что такое тестовая пирамида и почему она важна?

<details>
<summary>Ответ</summary>

Модель распределения тестов: много модульных (быстрые, дешёвые), меньше интеграционных, мало E2E (медленные, дорогие). Важна потому что: (1) быстрая обратная связь, (2) стабильность, (3) экономичность. Антипаттерн — перевёрнутая пирамида (много E2E) — ведёт к медленным, нестабильным тестам.

</details>

- [ ] Как проводить исследовательское тестирование?

<details>
<summary>Ответ</summary>

Метод SBTM: (1) **Хартия** — что исследуем, чем, зачем. (2) **Таймбокс** — 60-90 минут. (3) **Заметки** — находки по времени. (4) **Эвристики** — граничные значения, неожиданный ввод, прерывания, конкуренция. (5) **Отчёт** — баги, покрытие, области для дополнительного тестирования. Ключ: структурированное творчество, а не хаотичные клики.

</details>

- [ ] Как улучшать QA-процессы?

<details>
<summary>Ответ</summary>

Цикл: измерь → анализируй → предложи → внедри → проверь. Примеры: параллелизация CI (45→12 мин), карантин flaky-тестов (15→2%), shift-left (баги раньше). Важно: маленькие шаги, измеримые результаты, ретроспективы.

</details>

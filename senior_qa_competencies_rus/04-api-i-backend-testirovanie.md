## 1. REST API тестирование

### HTTP-методы и тестирование:

| Метод | Действие | Пример | Ожидаемый код |
|-------|---------|--------|--------------|
| **GET** | Чтение | GET /api/users/1 | 200 OK |
| **POST** | Создание | POST /api/users | 201 Created |
| **PUT** | Полное обновление | PUT /api/users/1 | 200 OK |
| **PATCH** | Частичное обновление | PATCH /api/users/1 | 200 OK |
| **DELETE** | Удаление | DELETE /api/users/1 | 204 No Content |

### Полная стратегия тестирования API:

```
1. Позитивные тесты — корректные данные → ожидаемый результат
2. Негативные тесты — невалидные данные → корректная ошибка
3. Аутентификация — без токена (401), чужая роль (403)
4. Граничные значения — min/max длины, пустые строки
5. Контрактные тесты — структура ответа соответствует схеме
6. Идемпотентность — повторный запрос не создаёт дублей
7. Обработка ошибок — понятные сообщения, без стектрейсов
```

---

## 2. HTTP-коды состояния

### Ключевые коды для тестирования:

| Код | Название | Когда проверять |
|-----|---------|----------------|
| **200** | OK | Успешный GET, PUT, PATCH |
| **201** | Created | Успешный POST, ресурс создан |
| **204** | No Content | Успешный DELETE |
| **400** | Bad Request | Невалидные входные данные |
| **401** | Unauthorized | Нет/невалидная аутентификация |
| **403** | Forbidden | Аутентифицирован, но нет прав |
| **404** | Not Found | Ресурс не существует |
| **409** | Conflict | Дублирование (напр., email) |
| **422** | Unprocessable Entity | Синтаксис верный, семантическая ошибка |
| **429** | Too Many Requests | Превышен лимит запросов |
| **500** | Internal Server Error | Ошибка сервера (не должно быть в happy path) |
| **503** | Service Unavailable | Сервис временно недоступен |

## 3. Валидация схемы ответа

### Пример валидации JSON-схемы:

```javascript
// Ожидаемая схема для GET /api/users/1
const userSchema = {
  type: "object",
  required: ["id", "name", "email", "createdAt"],
  properties: {
    id: { type: "number" },
    name: { type: "string", minLength: 1 },
    email: { type: "string", format: "email" },
    role: { type: "string", enum: ["user", "admin"] },
    createdAt: { type: "string", format: "date-time" }
  },
  additionalProperties: false  // Нет лишних полей
};

// В тесте проверяем ответ против схемы
expect(response.body).toMatchSchema(userSchema);
```

### Чек-лист валидации ответа:

- [ ] Все обязательные поля присутствуют
- [ ] Типы данных корректны (число, строка, массив)
- [ ] Нет лишних полей (утечка данных)
- [ ] Формат даты/времени стандартный (ISO 8601)
- [ ] Пустые коллекции — пустой массив `[]`, а не `null`
- [ ] Пагинация работает (limit, offset, totalCount)

## 4. Тестирование микросервисов

### Подходы к тестированию:

```
Уровни тестирования микросервисов:

1. Модульные тесты    — логика внутри одного сервиса
2. Контрактные тесты  — интерфейс между сервисами
3. Интеграционные     — взаимодействие 2-3 сервисов
4. E2E тесты          — полный пользовательский путь
```

### Контрактное тестирование:

```
Сервис A (потребитель) ←→ Сервис B (поставщик)

Потребитель определяет: "Мне нужно поле user.name типа string"
Поставщик гарантирует: "Я всегда возвращаю user.name типа string"

При изменении API поставщика — контрактный тест падает ДО деплоя
```

## 5. Инструменты API-тестирования

### Сравнение инструментов:

| Инструмент | Тип | Лучше для |
|-----------|-----|-----------|
| **Postman** | GUI + скрипты | Исследование, ручное тестирование |
| **REST Assured** | Java-библиотека | Автоматизация в Java-проектах |
| **SuperTest** | Node.js-библиотека | Автоматизация в JS/TS-проектах |
| **SoapUI** | GUI | SOAP + REST, корпоративные API |
| **k6** | CLI | Нагрузочное тестирование API |

### Пример теста в Postman:

```javascript
// Pre-request script
const timestamp = Date.now();
pm.environment.set("uniqueEmail", `test_${timestamp}@test.com`);

// Тело запроса POST /api/users
{
  "name": "Настя",
  "email": "{{uniqueEmail}}"
}

// Tests
pm.test("Статус 201", () => {
  pm.response.to.have.status(201);
});

pm.test("Тело содержит id", () => {
  const body = pm.response.json();
  pm.expect(body).to.have.property("id");
  pm.expect(body.name).to.equal("Настя");
});
```

## 6. Валидация данных в БД через SQL

### Типовые проверки после API-вызовов:

```sql
-- После POST /api/users
-- 1. Запись существует
SELECT * FROM users WHERE email = 'nastya@test.com';

-- 2. Нет дубликатов
SELECT COUNT(*) FROM users WHERE email = 'nastya@test.com';
-- Ожидание: ровно 1

-- 3. Связанные записи созданы
SELECT * FROM user_settings WHERE user_id = 42;

-- 4. После DELETE — soft delete
SELECT deleted_at FROM users WHERE id = 42;
-- Ожидание: NOT NULL

-- 5. После UPDATE — данные обновлены
SELECT name, updated_at FROM users WHERE id = 42;
-- Ожидание: name = 'Новое имя', updated_at > старого
```

## ✅ Вопросы для самопроверки

- [ ] Как тщательно протестировать REST API?

<details>
<summary>Ответ</summary>

Тестирование по нескольким измерениям: (1) **Функциональные** — коды, тело ответа, заголовки. (2) **Контрактные** — схема, обязательные поля, типы данных. (3) **Негативные** — невалидные данные, отсутствующие поля. (4) **Авторизация** — без токена (401), неправильная роль (403). (5) **Граничные** — min/max значения, пустые строки, unicode. (6) **Идемпотентность** — повторный запрос без дублей. (7) **Безопасность** — инъекции, CORS, rate limiting.

</details>

- [ ] Какие HTTP-коды важно проверять и почему?

<details>
<summary>Ответ</summary>

Ключевые: **200** — успешный GET/PUT. **201** — создан ресурс (POST). **204** — успешный DELETE. **400** — невалидные данные (с понятным сообщением). **401** — нет аутентификации. **403** — нет прав. **404** — ресурс не найден. **409** — конфликт (дублирование). **500** — серверная ошибка (не должна возникать при корректных данных). Важно проверять не только код, но и тело ответа, заголовки.

</details>

- [ ] Как тестировать микросервисы?

<details>
<summary>Ответ</summary>

Четыре уровня: (1) **Модульные** — логика внутри сервиса. (2) **Контрактные** — проверка интерфейсов между сервисами (Pact, Spring Cloud Contract). (3) **Интеграционные** — взаимодействие 2-3 сервисов. (4) **E2E** — полный путь пользователя. Контрактные тесты особенно важны — они обнаруживают несовместимые изменения API до деплоя.

</details>

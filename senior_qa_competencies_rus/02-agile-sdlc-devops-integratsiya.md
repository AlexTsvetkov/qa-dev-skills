## 1. Agile и Scrum для QA

### Роль QA в Scrum-церемониях:

| Церемония | Роль QA |
|-----------|---------|
| **Планирование спринта** | Оценка тестирования, выявление рисков, уточнение критериев приёмки |
| **Ежедневный стендап** | Статус тестирования, блокеры, координация с разработчиками |
| **Ревью спринта** | Демонстрация результатов тестирования, отчёт о качестве |
| **Ретроспектива** | Предложения по улучшению QA-процесса |
| **Уточнение бэклога** | Ревью историй, добавление тест-критериев |

### Definition of Done с точки зрения QA:

```markdown
## Definition of Done (QA-секция)
- [ ] Модульные тесты написаны и проходят
- [ ] API-тесты написаны для новых эндпоинтов
- [ ] Функциональное тестирование завершено
- [ ] Регрессионные тесты обновлены
- [ ] Кросс-браузерная проверка (Chrome, Firefox, Safari)
- [ ] Критические баги отсутствуют
- [ ] Тест-кейсы обновлены в системе управления тестами
```

### Написание критериев приёмки:

```
Формат Given-When-Then:

Пользовательская история: Как покупатель, я хочу применить скидочный код

  Given (Дано): Пользователь в корзине с товарами на $100
  When (Когда): Вводит действующий код "SAVE20"
  Then (Тогда): Общая сумма обновляется до $80
  And (И): Скидка отображается отдельной строкой

Негативный сценарий:
  Given: Пользователь в корзине
  When: Вводит истёкший код "EXPIRED"
  Then: Отображается сообщение "Промо-код истёк"
  And: Общая сумма не меняется
```

## 2. SDLC и роль QA

### QA на каждом этапе SDLC:

```
Требования → Дизайн → Разработка → Тестирование → Деплой → Поддержка
    ↑            ↑          ↑           ↑           ↑          ↑
  Ревью      Ревью     Модульные   Функцион.  Смоук-тест  Мониторинг
 требований  дизайна    тесты     тестирование  в проде    в проде
```

### Модели SDLC:

| Модель | QA-подход | Лучше для |
|--------|-----------|-----------|
| **Waterfall** | Тестирование после разработки | Фиксированные требования |
| **Agile/Scrum** | Непрерывное тестирование в спринтах | Эволюционирующие продукты |
| **Kanban** | Тестирование по потоку | Поддержка, исправления |
| **DevOps** | Автоматизация на каждом этапе | Непрерывная поставка |

## 3. Культура DevOps и непрерывное тестирование

### Принципы непрерывного тестирования:

```
Код → Сборка → Тест → Деплой → Мониторинг
        ↑        ↑       ↑          ↑
    Модульные  API/    Смоук     Алерты
    тесты    интегр.   тесты   и метрики
```

### Shift-Left: перенос тестирования на ранние этапы

| Традиционный подход | Shift-Left подход |
|-------------------|-------------------|
| Тестирование после разработки | Тестирование начинается с требований |
| Баги находят поздно (дорого) | Баги находят рано (дёшево) |
| QA отдельно от разработки | QA интегрировано в команду |
| Ручная регрессия | Автоматизированная обратная связь |

### Активности Shift-Left:

```
1. ТРЕБОВАНИЯ: Ревью пользовательских историй на тестируемость
2. ДИЗАЙН: Участие в обсуждении архитектуры (тестируемость API)
3. РАЗРАБОТКА: Ревью модульных тестов, парное тестирование
4. ИНТЕГРАЦИЯ: Автоматизированные API-тесты в CI
5. ДЕПЛОЙ: Автоматический смоук-тест после деплоя
```

## 4. Интеграция тестов в CI/CD

### Стратегия тестирования в пайплайне:

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────┐
│   Коммит     │──→│  Модульные   │──→│ API / Интегр.│──→│   E2E    │
│              │   │  тесты       │   │   тесты      │   │  тесты   │
│              │   │  (~2 мин)    │   │  (~10 мин)   │   │ (~20 мин)│
└──────────────┘   └──────────────┘   └──────────────┘   └──────────┘
                        ↓ фейл            ↓ фейл           ↓ фейл
                    Уведомление       Уведомление       Уведомление
                    + блокировка      + блокировка      + отчёт
```

### Пример GitHub Actions:

```yaml
name: Тестовый пайплайн
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  api-tests:
    needs: unit-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:api

  e2e-tests:
    needs: api-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npx playwright install --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: test-report
          path: playwright-report/
```

### Сравнение CI/CD инструментов:

| Инструмент | Конфигурация | Лучше для |
|-----------|-------------|-----------|
| **GitHub Actions** | YAML | GitHub-репозитории |
| **Jenkins** | Groovy/YAML | Корпоративные, сложные пайплайны |
| **GitLab CI** | YAML | GitLab-репозитории |
| **Azure DevOps** | YAML | Экосистема Microsoft |

## ✅ Вопросы для самопроверки

- [ ] Какова роль QA в Scrum-церемониях?

<details>
<summary>Ответ</summary>

QA участвует в каждой церемонии: **Планирование** — оценка тестирования, выявление рисков. **Стендап** — статус тестирования, блокеры. **Ревью** — демонстрация качества, метрики. **Ретро** — предложения по улучшению процесса. **Уточнение бэклога** — ревью историй на тестируемость, добавление критериев приёмки.

</details>

- [ ] Что такое shift-left и как QA его реализует?

<details>
<summary>Ответ</summary>

Shift-left — перенос тестирования на более ранние этапы SDLC. Реализация: (1) Ревью требований на тестируемость. (2) Участие в обсуждении архитектуры. (3) Ревью модульных тестов разработчиков. (4) Автоматизация API-тестов в CI. (5) Быстрая обратная связь через автоматизацию. Цель — находить дефекты рано, когда исправление стоит в 10 раз дешевле.

</details>

- [ ] Как интегрировать тесты в CI/CD пайплайн?

<details>
<summary>Ответ</summary>

Следуя пирамиде тестирования: (1) **Модульные тесты** — на каждый коммит (быстро, <5 мин). (2) **API/интеграционные** — после модульных (<15 мин). (3) **E2E** — на основную ветку или перед релизом (<30 мин). Практики: быстрый отказ, загрузка артефактов при ошибках, уведомления о сбоях, параллелизация, защита веток (PR не мёржится при падении тестов).

</details>

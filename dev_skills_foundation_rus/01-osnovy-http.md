## 1. Что такое HTTP?

**HTTP (HyperText Transfer Protocol)** — это основа передачи данных в вебе. Он определяет, как сообщения форматируются и передаются между **клиентом** (обычно браузером или приложением) и **сервером**.

### Как это работает (упрощённо):

```
Клиент (Браузер)                    Сервер
     |                                |
     |  ---- HTTP-запрос --->         |
     |                                |  (обрабатывает запрос)
     |  <--- HTTP-ответ ----          |
     |                                |
```

1. **Клиент** отправляет **HTTP-запрос** серверу (например, «Дай мне главную страницу»)
2. **Сервер** обрабатывает запрос и отправляет обратно **HTTP-ответ** (например, HTML страницы)

### Ключевые характеристики HTTP:
- **Без сохранения состояния (Stateless)** — каждый запрос независим; сервер не помнит предыдущие запросы
- **Текстовый** — сообщения читаемы человеком (по крайней мере в HTTP/1.1)
- **Модель запрос-ответ** — клиент всегда инициирует, сервер всегда отвечает

### HTTP vs HTTPS

**HTTPS** = HTTP + **шифрование TLS/SSL**. Данные шифруются при передаче, поэтому никто не может их прочитать. Сегодня почти все сайты используют HTTPS.

## 2. Структура HTTP-запроса

Каждый HTTP-запрос состоит из следующих частей:

```
[Метод] [URL] [Версия HTTP]
[Заголовки]

[Тело (необязательно)]
```

### Реальный пример:

```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{
  "name": "Настя",
  "role": "QA Engineer"
}
```

Разберём по частям:

| Часть | Пример | Описание |
|-------|--------|----------|
| **Метод** | `POST` | Какое действие выполнить |
| **URL/Путь** | `/api/users` | Куда отправить запрос |
| **Версия HTTP** | `HTTP/1.1` | Версия протокола |
| **Заголовки** | `Content-Type: application/json` | Метаданные о запросе |
| **Тело** | `{"name": "Настя", ...}` | Данные, отправляемые с запросом |

## 3. Структура HTTP-ответа

```
[Версия HTTP] [Код состояния] [Сообщение состояния]
[Заголовки]

[Тело]
```

### Реальный пример:

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/users/42

{
  "id": 42,
  "name": "Настя",
  "role": "QA Engineer",
  "createdAt": "2024-01-15T10:30:00Z"
}
```

| Часть | Пример | Описание |
|-------|--------|----------|
| **Код состояния** | `201` | Числовой результат запроса |
| **Сообщение** | `Created` | Человекочитаемое описание статуса |
| **Заголовки** | `Content-Type: application/json` | Метаданные об ответе |
| **Тело** | `{"id": 42, ...}` | Данные, возвращённые сервером |

## 4. HTTP-методы

HTTP-методы (также называемые «глаголами») говорят серверу, **какое действие** нужно выполнить.

### Основные методы:

| Метод | Назначение | Есть тело? | Безопасный? | Идемпотентный? |
|-------|-----------|-----------|------------|----------------|
| **GET** | Получить данные | Нет | ✅ Да | ✅ Да |
| **POST** | Создать новые данные | Да | ❌ Нет | ❌ Нет |
| **PUT** | Заменить данные полностью | Да | ❌ Нет | ✅ Да |
| **PATCH** | Обновить данные частично | Да | ❌ Нет | ❌ Нет |
| **DELETE** | Удалить данные | Опционально | ❌ Нет | ✅ Да |
| **HEAD** | Как GET, но без тела | Нет | ✅ Да | ✅ Да |
| **OPTIONS** | Проверить доступные методы | Нет | ✅ Да | ✅ Да |

### Важные термины:

- **Безопасный (Safe)** — запрос НЕ изменяет данные на сервере (GET, HEAD, OPTIONS)
- **Идемпотентный (Idempotent)** — вызов один или несколько раз даёт одинаковый результат (GET, PUT, DELETE)

### Примеры на практике:

```
GET    /api/users          → Получить список всех пользователей
GET    /api/users/42       → Получить пользователя с ID 42
POST   /api/users          → Создать нового пользователя
PUT    /api/users/42       → Заменить пользователя 42 полностью
PATCH  /api/users/42       → Обновить некоторые поля пользователя 42
DELETE /api/users/42       → Удалить пользователя 42
```

### 🤔 PUT vs PATCH — в чём разница?

**PUT** заменяет весь ресурс:
```json
// PUT /api/users/42
// Нужно отправить ВСЕ поля
{
  "name": "Настя",
  "role": "Senior QA",
  "email": "nastya@example.com"
}
```

**PATCH** обновляет только определённые поля:
```json
// PATCH /api/users/42
// Отправляем только то, что хотим изменить
{
  "role": "Senior QA"
}
```

## 5. HTTP-заголовки

Заголовки — это **пары ключ-значение**, которые несут метаданные о запросе или ответе. Они как этикетки на посылке — описывают, что внутри и как с этим обращаться.

### Распространённые заголовки запроса:

| Заголовок | Назначение | Пример |
|-----------|-----------|--------|
| `Host` | Доменное имя сервера | `Host: api.example.com` |
| `Content-Type` | Формат данных в теле | `Content-Type: application/json` |
| `Accept` | Какой формат клиент хочет получить | `Accept: application/json` |
| `Authorization` | Учётные данные аутентификации | `Authorization: Bearer <token>` |
| `User-Agent` | Информация о клиенте | `User-Agent: Mozilla/5.0...` |
| `Cookie` | Куки, отправляемые серверу | `Cookie: session=abc123` |
| `Cache-Control` | Инструкции кэширования | `Cache-Control: no-cache` |

### Распространённые заголовки ответа:

| Заголовок | Назначение | Пример |
|-----------|-----------|--------|
| `Content-Type` | Формат тела ответа | `Content-Type: application/json` |
| `Content-Length` | Размер тела ответа в байтах | `Content-Length: 348` |
| `Set-Cookie` | Установить куку на клиенте | `Set-Cookie: session=abc123` |
| `Location` | URL для редиректов или новых ресурсов | `Location: /api/users/42` |
| `Access-Control-Allow-Origin` | CORS — кто может получить доступ | `Access-Control-Allow-Origin: *` |
| `X-Request-Id` | Уникальный ID для отслеживания | `X-Request-Id: abc-123-def` |

### Часто встречающиеся значения Content-Type:

| Content-Type | Описание |
|-------------|----------|
| `application/json` | JSON-данные (самый частый для API) |
| `application/xml` | XML-данные |
| `text/html` | HTML-страница |
| `text/plain` | Обычный текст |
| `multipart/form-data` | Загрузка файлов |
| `application/x-www-form-urlencoded` | Данные HTML-формы |

## 6. Коды состояния HTTP

Коды состояния сообщают, **что произошло** с вашим запросом. Они делятся на 5 категорий:

### Категории:

| Диапазон | Категория | Значение |
|----------|----------|----------|
| **1xx** | Информационные | Запрос получен, обработка продолжается... |
| **2xx** | Успех | Запрос выполнен успешно |
| **3xx** | Перенаправление | Нужно перейти в другое место |
| **4xx** | Ошибка клиента | Вы (клиент) допустили ошибку |
| **5xx** | Ошибка сервера | Сервер не справился |

### Самые важные коды состояния:

#### ✅ 2xx — Успех

| Код | Название | Значение |
|-----|---------|----------|
| **200** | OK | Запрос выполнен успешно (общий успех) |
| **201** | Created | Создан новый ресурс (после POST) |
| **204** | No Content | Успех, но нечего возвращать (часто после DELETE) |

#### 🔀 3xx — Перенаправление

| Код | Название | Значение |
|-----|---------|----------|
| **301** | Moved Permanently | Ресурс навсегда перемещён на новый URL |
| **302** | Found | Ресурс временно по другому URL |
| **304** | Not Modified | Кэшированная версия всё ещё актуальна |

#### ❌ 4xx — Ошибки клиента

| Код | Название | Значение |
|-----|---------|----------|
| **400** | Bad Request | Некорректный запрос (например, невалидный JSON) |
| **401** | Unauthorized | Не аутентифицирован (нужно войти) |
| **403** | Forbidden | Аутентифицирован, но нет прав доступа |
| **404** | Not Found | Ресурс не существует |
| **405** | Method Not Allowed | HTTP-метод не поддерживается для этого URL |
| **409** | Conflict | Конфликт с текущим состоянием (например, дубликат) |
| **422** | Unprocessable Entity | Запрос валиден, но логически некорректен |
| **429** | Too Many Requests | Превышен лимит запросов — подождите! |

#### 💥 5xx — Ошибки сервера

| Код | Название | Значение |
|-----|---------|----------|
| **500** | Internal Server Error | Общая ошибка сервера |
| **502** | Bad Gateway | Сервер получил невалидный ответ от вышестоящего сервера |
| **503** | Service Unavailable | Сервер временно недоступен |
| **504** | Gateway Timeout | Вышестоящий сервер не ответил вовремя |

### 🧠 Взгляд QA-инженера: на что обращать внимание

Как QA-инженер, обращайте внимание на:
- **Возвращает ли API правильный код состояния?** (например, 201 для создания, а не 200)
- **Консистентны ли ответы об ошибках?** (например, всегда возвращать JSON-объект ошибки)
- **Правильно ли работает 401 vs 403?** (неавторизован vs запрещено — это разные вещи!)
- **Что происходит при 5xx?** — Обрабатывает ли клиент ошибки сервера корректно?

## 7. Структура URL

Понимание URL необходимо для тестирования API:

```
https://api.example.com:443/api/v2/users?role=qa&active=true#section1
\___/   \______________/ \_/ \_____________/ \___________________/ \______/
  |           |           |        |                  |               |
схема       хост        порт     путь        параметры запроса    фрагмент
```

| Часть | Описание | Пример |
|-------|----------|--------|
| **Схема** | Протокол | `https` |
| **Хост** | Адрес сервера | `api.example.com` |
| **Порт** | Сетевой порт (необязательно) | `443` (по умолчанию для HTTPS) |
| **Путь** | Расположение ресурса | `/api/v2/users` |
| **Параметры запроса** | Фильтры/опции | `?role=qa&active=true` |
| **Фрагмент** | Якорь на стороне клиента | `#section1` |

### Параметры запроса (Query Parameters)

Параметры запроса используются для фильтрации, сортировки или пагинации данных:

```
GET /api/users?role=qa&active=true&page=1&limit=20&sort=name
```

| Параметр | Назначение |
|----------|-----------|
| `role=qa` | Фильтр по роли |
| `active=true` | Фильтр по статусу |
| `page=1` | Пагинация — какая страница |
| `limit=20` | Пагинация — сколько на страницу |
| `sort=name` | Сортировка по имени |

### Упражнение 4: Практика с `curl`

Используйте командную строку для выполнения реальных HTTP-запросов:

```bash
# 1. Простой GET-запрос — посмотрите ответ от публичного API
curl -v https://httpbin.org/get

# 2. POST-запрос с JSON-телом
curl -v -X POST https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name": "Настя", "skill": "QA"}'

# 3. Посмотреть только заголовки ответа
curl -I https://httpbin.org/get

# 4. PUT-запрос
curl -v -X PUT https://httpbin.org/put \
  -H "Content-Type: application/json" \
  -d '{"name": "Настя", "skill": "Senior QA"}'

# 5. DELETE-запрос
curl -v -X DELETE https://httpbin.org/delete
```

**Флаг `-v`** (verbose) показывает полный запрос и ответ, включая заголовки. Это очень полезно для обучения!

**Изучите вывод и определите:**
- Заголовки запроса, отправленные curl
- Код состояния ответа
- Заголовки ответа
- Тело ответа

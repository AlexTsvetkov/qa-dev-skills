## 1. Тест-стратегия vs. Тест-план

```
  Тест-стратегия (ВЫСОКОУРОВНЕВАЯ)
  ┌────────────────────────────────────────────┐
  │ ЧТО тестируем и ПОЧЕМУ                    │
  │ • Масштаб и подход к тестированию          │
  │ • Типы тестирования                        │
  │ • Стратегия автоматизации                  │
  │ • Управление рисками                       │
  │ • Инструменты и окружения                  │
  │ • Критерии входа/выхода                    │
  │ Обычно: 1 на проект/продукт               │
  └────────────────────────────────────────────┘

  Тест-план (ДЕТАЛЬНЫЙ)
  ┌────────────────────────────────────────────┐
  │ КАК тестируем и КОГДА                      │
  │ • Конкретные тестовые задачи               │
  │ • Расписание и вехи                        │
  │ • Распределение ресурсов                   │
  │ • Конкретные тест-кейсы                    │
  │ • Зависимости и риски                      │
  │ Обычно: 1 на релиз/спринт/фичу            │
  └────────────────────────────────────────────┘

  Стратегия → определяет подход
  План     → реализует подход для конкретного релиза
```

## 2. Жизненный цикл тестирования (STLC)

```
  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
  │ 1. Анализ    │──→│ 2. Планиро-  │──→│ 3. Дизайн    │
  │ требований   │   │ вание тестов │   │ тест-кейсов  │
  └──────────────┘   └──────────────┘   └──────────────┘
         │                                      │
         ▼                                      ▼
  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
  │ 6. Закрытие  │←──│ 5. Отчёт-    │←──│ 4. Выполне-  │
  │ цикла        │   │ ность        │   │ ние тестов   │
  └──────────────┘   └──────────────┘   └──────────────┘

  1. Анализ требований
     - Обзор требований, историй, критериев приёмки
     - Определение тестируемых элементов
     - Сессии 3 Amigos (Dev + QA + PO)

  2. Планирование
     - Оценка трудозатрат
     - Подготовка окружений
     - Распределение задач

  3. Дизайн тест-кейсов
     - Написание тест-кейсов
     - Подготовка тестовых данных
     - Обзор и утверждение

  4. Выполнение
     - Прогон тестов (ручной + автоматизация)
     - Логирование багов
     - Регрессионное тестирование

  5. Отчётность
     - Статус тестирования
     - Метрики и дашборды
     - Оценка критериев выхода

  6. Закрытие цикла
     - Ретроспектива
     - Архивирование артефактов
     - Извлечённые уроки
```

## 3. Матрица трассировки (RTM)

```
  Связывает: Требование → Тест-кейс → Баг → Статус

  ┌──────────┬──────────────────┬────────────┬─────────┬────────┐
  │ Req ID   │ Описание         │ Тест-кейсы │ Статус  │ Баги   │
  ├──────────┼──────────────────┼────────────┼─────────┼────────┤
  │ REQ-001  │ Логин по email   │ TC-01,02,03│ Покрыто │ BUG-12 │
  │ REQ-002  │ Сброс пароля     │ TC-04,05   │ Покрыто │ —      │
  │ REQ-003  │ Роли и доступ    │ —          │ Разрыв! │ —      │
  │ REQ-004  │ Профиль          │ TC-08      │ Покрыто │ BUG-15 │
  └──────────┴──────────────────┴────────────┴─────────┴────────┘

  REQ-003 = РАЗРЫВ в покрытии! Нужны тест-кейсы.

  Зачем нужна RTM:
    ✓ Выявление непокрытых требований
    ✓ Оценка покрытия тестами
    ✓ Анализ влияния при изменении требований
    ✓ Отслеживание связи баг → требование
```

## 4. Планирование и оценка тестирования

```
  Техники оценки:
  ┌─────────────────────────────────────────────────────┐
  │ 1. На основе опыта                                 │
  │    - Экспертная оценка по прошлым проектам          │
  │    - Быстро, но субъективно                        │
  │                                                     │
  │ 2. На основе тест-кейсов                           │
  │    - Подсчёт тест-кейсов × среднее время           │
  │    - 150 кейсов × 15 мин = 2250 мин ≈ 37.5 часов  │
  │                                                     │
  │ 3. На основе рисков                                │
  │    - Приоритет = Вероятность × Влияние              │
  │    - Больше усилий на высокий риск                  │
  │                                                     │
  │ 4. Planning Poker (Agile)                           │
  │    - Команда оценивает в story points               │
  │    - Согласованная оценка                           │
  └─────────────────────────────────────────────────────┘

  Матрица приоритизации:
  ┌────────────────────┬──────────┬──────────┐
  │                    │ Высокое  │ Низкое   │
  │                    │ влияние  │ влияние  │
  ├────────────────────┼──────────┼──────────┤
  │ Высокая вероятность│ КРИТИЧНО │ СРЕДНИЙ  │
  │ Низкая вероятность │ СРЕДНИЙ  │ НИЗКИЙ   │
  └────────────────────┴──────────┴──────────┘
```

## 5. Метрики качества и KPI

```
  Покрытие тестами:
    Покрытие = (Покрытые требования / Всего требований) × 100%
    Цель: ≥ 95% для критичных функций

  Тренды дефектов:
    - Найдено vs. Исправлено по спринтам
    - Процент повторного открытия багов
    - Плотность дефектов (дефекты / модуль)

  Здоровье релиза:
    - Открытые критичные/блокирующие баги = 0
    - Пройдено тестов: ≥ 95%
    - Все критерии выхода выполнены

  Эффективность тестирования:
    = Баги найдены QA / (QA + Продакшен баги) × 100%
    Цель: ≥ 90% (ловим баги ДО продакшена)
```

## ✅ Вопросы для самопроверки

1. В чём разница между тест-стратегией и тест-планом?

<details>
<summary>Ответ</summary>

**Тест-стратегия** — высокоуровневый документ, определяющий общий подход к тестированию: ЧТО тестируем, ПОЧЕМУ, какие типы тестирования используем, какие инструменты и окружения. Обычно один на проект. **Тест-план** — детальный документ, описывающий КАК и КОГДА тестируем: конкретные задачи, расписание, ресурсы, тест-кейсы. Обычно один на релиз или спринт. Стратегия определяет подход, план — его реализует.

</details>

2. Что такое Матрица трассировки требований и зачем она нужна?

<details>
<summary>Ответ</summary>

**RTM (Requirements Traceability Matrix)** — таблица, связывающая требования с тест-кейсами, багами и статусом покрытия. Нужна для: (1) выявления непокрытых требований (разрывов), (2) оценки полноты покрытия тестами, (3) анализа влияния при изменении требований, (4) отслеживания связи между багами и требованиями.

</details>

3. Перечислите шесть этапов жизненного цикла тестирования (STLC).

<details>
<summary>Ответ</summary>

(1) **Анализ требований** — обзор требований, определение объёма тестирования. (2) **Планирование** — оценка трудозатрат, подготовка окружений, распределение задач. (3) **Дизайн тест-кейсов** — написание кейсов, подготовка данных. (4) **Выполнение тестов** — прогон тестов, логирование багов. (5) **Отчётность** — статус, метрики, дашборды. (6) **Закрытие цикла** — ретроспектива, архивирование, извлечённые уроки.

</details>

4. Какие метрики качества наиболее важны для оценки здоровья релиза?

<details>
<summary>Ответ</summary>

Ключевые метрики: (1) **Покрытие тестами** — % покрытых требований (цель ≥ 95%), (2) **Открытые критичные баги** — должно быть 0 для релиза, (3) **Процент прохождения тестов** — цель ≥ 95%, (4) **Эффективность тестирования** — % багов, найденных до продакшена (цель ≥ 90%), (5) **Тренды дефектов** — найдено vs исправлено, повторные открытия.

</details>

5. Объясните подход к тестированию на основе рисков.

<details>
<summary>Ответ</summary>

**Тестирование на основе рисков** приоритизирует усилия по формуле: Приоритет = Вероятность × Влияние. Высокий риск (высокая вероятность + большое влияние) получает больше тестового покрытия и внимания. Низкий риск может быть покрыт минимально или отложен. Это позволяет оптимально распределять ресурсы, когда невозможно протестировать всё.

</details>

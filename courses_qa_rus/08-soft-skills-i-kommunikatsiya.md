## 1. Документация и тестовые артефакты

### Написание эффективных баг-репортов:

```markdown
## BUG-1234: Оплата не проходит для карт с 3D Secure

**Серьёзность**: Критическая | **Приоритет**: P1

### Окружение
- Браузер: Chrome 120, Windows 11
- Среда: Staging (v3.2.1)

### Шаги воспроизведения
1. Войти как user@test.com
2. Добавить товар в корзину
3. Перейти к оплате
4. Ввести карту с 3D Secure (4000 0000 0000 3220)
5. Подтвердить 3D Secure в popup

### Ожидаемый результат
Оплата успешна, отображается страница подтверждения.

### Фактический результат
После 3D Secure popup — белый экран. Консоль: 
"TypeError: Cannot read property 'status' of undefined"

### Влияние
~30% карт используют 3D Secure → блокирует оплату для трети пользователей.

### Вложения
- [Скриншот белого экрана]
- [Лог консоли]
- [HAR-файл сетевых запросов]
```

### Шаблон статуса тестирования:

```markdown
## Статус тестирования — Спринт 24 (15.03.2026)

### Прогресс
| Метрика | Значение |
|---------|---------|
| Тест-кейсы: всего | 245 |
| Выполнено | 210 (86%) |
| Успешно | 195 (93%) |
| Упало | 12 (5.7%) |
| Заблокировано | 3 (1.4%) |

### Критические находки
1. **BUG-1234** (Критич.) — 3D Secure не работает
2. **BUG-1235** (Высок.) — Таймаут при 100+ товарах в корзине

### Риски
- 3D Secure блокирует ~30% оплат → нужен фикс до релиза
- Производительность корзины — обходной путь есть

### Рекомендация
Условный GO после фикса BUG-1234.
```

## 2. Лидерство и командное взаимодействие

### Принципы QA-лидерства:

| Принцип | Пример действия |
|---------|----------------|
| **Влияние без полномочий** | Убеждать данными, не должностью |
| **Проактивность** | Предлагать улучшения, не ждать указаний |
| **Прозрачность** | Делиться плохими новостями рано |
| **Поддержка** | Помогать команде расти |
| **Ответственность** | Владеть качеством, не только тестированием |

### Управление конфликтами:

```
Ситуация: Разработчик не согласен, что это баг.

1. ВЫСЛУШАЙ — пойми позицию разработчика
2. СОБЕРИ ДАННЫЕ — скриншоты, спеки, аналогичные страницы
3. ПОКАЖИ ФАКТЫ — "На 3 других страницах элемент слева"
4. ПРИВЛЕКИ ЭКСПЕРТА — если нужно (дизайнер, PO)
5. ПРИМИ РЕШЕНИЕ — на основе данных, не мнений

Ключ: спор о продукте, НЕ о людях.
```

## 3. Решение проблем и критическое мышление

### Фреймворк анализа проблем:

```
1. ОПРЕДЕЛИ — Что именно не работает?
   "Оплата падает для 3% пользователей"

2. СОБЕРИ ДАННЫЕ — Когда, где, для кого?
   "Только карты с 3D Secure, с 12 марта"

3. ИЗОЛИРУЙ — Сузь область
   "Только staging, прод работает → деплой 12 марта сломал"

4. ПРОВЕРЬ ГИПОТЕЗУ — Тестируй предположение
   "Откат деплоя → проблема исчезает → подтверждено"

5. РЕШИ — Определи корневую причину
   "Новая версия SDK платёжного шлюза несовместима"
```

### 5 Почему (Root Cause Analysis):

```
Проблема: Тесты упали в CI

Почему 1: E2E-тест корзины не прошёл
Почему 2: Кнопка "Оплатить" не найдена
Почему 3: Селектор изменился после обновления UI
Почему 4: Разработчик изменил CSS-классы
Почему 5: Тесты используют CSS-селекторы вместо data-testid

Решение: Перевести селекторы на data-testid + добавить
в Definition of Done проверку тестовых селекторов
```

## 4. Согласование со стейкхолдерами

### Адаптация коммуникации:

| Аудитория | Что говорить | Как говорить |
|-----------|-------------|-------------|
| **Разработчики** | Технические детали, стектрейсы | Чётко, с кодом и логами |
| **PM/PO** | Влияние на пользователей, сроки | Бизнес-языком |
| **Руководство** | Риски, KPI, рекомендации | Кратко, с цифрами |
| **Клиенты** | Статус, ожидания | Простым языком |

### Go/No-Go решение:

```markdown
## Go/No-Go: Релиз 3.2.0

### ✅ GO факторы
- 95% тестов успешно
- Критический путь (регистрация → оплата) работает
- Производительность в пределах SLA

### ⚠️ Известные проблемы
- BUG-1235: Медленная корзина при 100+ товаров
  - Влияние: <2% пользователей
  - Обходной путь: есть
  - Фикс: следующий спринт

### Рекомендация: УСЛОВНЫЙ GO
- Релиз с усиленным мониторингом
- Hotfix для BUG-1235 готов через 48ч
```

## 5. Упрощение сложных концепций

### Техники объяснения:

| Техника | Пример |
|---------|--------|
| **Аналогия** | "API как официант: принимает заказ, несёт на кухню, приносит блюдо" |
| **Визуализация** | Схемы, диаграммы вместо текста |
| **Конкретные числа** | "Затронуто 30% пользователей" вместо "много" |
| **Примеры** | "Как если бы вы заказали пиццу, а привезли суши" |
| **Уровни детализации** | Краткое резюме + детали по запросу |

### Примеры упрощения:

```
Техническое: "Race condition в микросервисе приводит 
к inconsistent state при concurrent writes"

Для PM: "Когда два пользователя одновременно меняют одни 
данные, один из них может потерять свои изменения"

Для руководства: "Есть риск потери данных при пиковой 
нагрузке. Затрагивает ~3% операций. Фикс: 2 дня."
```

## ✅ Вопросы для самопроверки

- [ ] Как написать эффективный баг-репорт?

<details>
<summary>Ответ</summary>

Структура: (1) **Заголовок** — чётко описывает проблему. (2) **Серьёзность/приоритет**. (3) **Окружение** — браузер, версия, среда. (4) **Шаги** — точные, воспроизводимые. (5) **Ожидаемый vs фактический** результат. (6) **Влияние** — кого затрагивает, как сильно. (7) **Вложения** — скриншоты, логи, видео. Ключ: разработчик должен воспроизвести баг без дополнительных вопросов.

</details>

- [ ] Как адаптировать коммуникацию для разных аудиторий?

<details>
<summary>Ответ</summary>

**Разработчикам**: технические детали, код, логи. **PM/PO**: влияние на пользователей, сроки, приоритеты. **Руководству**: цифры, риски, варианты решений, рекомендация. **Клиентам**: простой язык, статус, ожидания. Принцип: начинай с резюме, детали по запросу. Всегда количественно оценивай влияние.

</details>

- [ ] Как разрешать конфликты с разработчиками?

<details>
<summary>Ответ</summary>

(1) **Выслушать** — понять позицию. (2) **Собрать данные** — факты вместо мнений. (3) **Показать доказательства** — спеки, скриншоты, аналогичные примеры. (4) **Привлечь эксперта** если нужно. (5) **Фокус на продукте** — спорим о качестве продукта, не о людях. Ключ: решения на основе данных и общих целей.

</details>

- [ ] Как упрощать технические концепции для нетехнической аудитории?

<details>
<summary>Ответ</summary>

Техники: (1) **Аналогии** из повседневной жизни. (2) **Конкретные числа** вместо абстрактных описаний. (3) **Визуализация** — схемы, диаграммы. (4) **Примеры** — конкретные сценарии. (5) **Уровни** — краткое резюме + детали по запросу. Избегать жаргона. Фокус на влиянии и действиях, а не на технических причинах.

</details>

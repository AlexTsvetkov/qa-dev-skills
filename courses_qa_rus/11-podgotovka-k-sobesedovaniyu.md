## 1. Стратегия и лидерство

### В: Как вы разрабатываете тест-стратегию?

```
Ответ:
1. Анализирую требования и архитектуру
2. Определяю области риска (бизнес + технические)
3. Выбираю уровни тестирования (unit, API, E2E)
4. Определяю инструменты и фреймворки
5. Устанавливаю метрики и критерии выхода
6. Планирую ресурсы и сроки
7. Согласовываю с командой и стейкхолдерами

Пример: "Для микросервисного проекта — упор на API-тесты (60%),
контрактные тесты между сервисами, минимум E2E для критических путей."
```

### В: Какие QA-метрики вы отслеживаете?

```
- Покрытие требований (>95%)
- Плотность дефектов (баги / модуль)
- Процент прохождения тестов (>95%)
- Баги в проде (цель: 0 критических)
- Flaky rate (<5%)
- Время CI-пайплайна (<15 мин)
```

### В: Как вы определяете риски и приоритеты?

```
Матрица: Вероятность × Влияние = Приоритет

  Оплата + высокая вероятность = P1 (тестируем первым)
  Настройки профиля + низкая вероятность = P3

Критерии выхода:
  - 100% P1 тестов пройдено
  - 0 критических/блокирующих багов
  - >90% P2 тестов пройдено
```

## 2. Инженерия автоматизации

### В: Как вы проектируете фреймворк?

```
Архитектура:
  ├── Page Objects / API Clients
  ├── Тестовые данные (фабрики, фикстуры)
  ├── Утилиты (ожидания, ретраи, логирование)
  ├── Конфигурация (среды, браузеры)
  └── Отчёты (Allure, HTML)

Принципы: DRY, независимость тестов, читаемость, CI/CD с первого дня
```

### В: Как вы боретесь с flaky-тестами?

```
1. Обнаружение — отслеживаю % нестабильных
2. Карантин — изолирую в отдельный сьют
3. Анализ причин:
   - Гонки → явные ожидания
   - Данные → изоляция
   - Внешние зависимости → моки
4. Починка и возврат
5. Политика: flaky > 3 раз → блокирует мерж
```

### В: Как вы интегрируете тесты в CI/CD?

```
Пайплайн:
  Коммит → Линтинг → Unit-тесты → API-тесты → E2E → Деплой

Принципы:
  - Быстрые тесты первыми
  - Параллельное выполнение
  - Результаты в Slack/Teams
  - Блокировка мержа при падении
```

## 3. Техническая глубина

### В: Как вы тестируете REST API?

```
1. Контрактное — схема, типы данных
2. Функциональное — CRUD операции
3. Негативное — 400, 401, 403, 404, 500
4. Граничные значения — пустые, max, спецсимволы
5. Производительность — время отклика, нагрузка
6. Безопасность — инъекции, авторизация
```

### В: Напишите SQL для поиска дубликатов:

```sql
SELECT email, COUNT(*) as cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

### В: Объясните техники тест-дизайна:

```
- Граничные значения: min-1, min, max, max+1
- Эквивалентное разбиение: классы с одинаковым поведением
- Таблицы решений: комбинации условий → действия
- Диаграммы состояний: переходы между состояниями
- Попарное тестирование: комбинации параметров
```

## 4. AI-тестирование

### В: Как тестировать AI/LLM функции?

```
1. Точность ответов — сравнение с эталоном
2. Галлюцинации — проверка фактов
3. Заземлённость — ответ основан на данных
4. Консистентность — одинаковые вопросы = похожие ответы
5. Безопасность — нет вредного контента
6. Граничные случаи — пустой ввод, противоречивые запросы
```

### В: Какие метрики AI вы используете?

```
- Hallucination Rate: % ответов с выдуманными фактами
- Grounding Score: % ответов основанных на данных
- Consistency Score: совпадение ответов при перефразировании
- Latency P95: время отклика для 95% запросов
- Safety Rate: % безопасных ответов
```

## 5. Коммуникация и лидерство

### В: Как вы сообщаете о рисках стейкхолдерам?

```
Формат:
  ЧТО: Описание риска
  ВЛИЯНИЕ: На что влияет (пользователи, доход, репутация)
  ВЕРОЯТНОСТЬ: Высокая/Средняя/Низкая
  ВАРИАНТЫ:
    A) Исправить сейчас — задержка 2 дня
    B) Релиз с обходным путём — риск X%
    C) Отложить фичу — нет риска
  РЕКОМЕНДАЦИЯ: Вариант A, потому что...

Ключ: факты, а не эмоции. Данные, а не мнения.
```

### В: Как вы менторите junior QA?

```
Подход:
1. Парное тестирование — показываю мышление
2. Код-ревью — конструктивная обратная связь
3. Постепенная сложность — от простых задач к сложным
4. Документация — шаблоны и чеклисты
5. 1-on-1 — регулярные встречи для обсуждения роста
```

## 6. Чек-лист подготовки к собеседованию

### Перед собеседованием:

```
□ Изучить компанию и продукт
□ Подготовить STAR-ответы (Ситуация, Задача, Действие, Результат)
□ Подготовить примеры из опыта для каждой области
□ Повторить SQL (JOIN, GROUP BY, подзапросы)
□ Повторить API-тестирование (статус-коды, валидация)
□ Подготовить вопросы к интервьюеру
```

### Формат STAR для ответов:

```
Вопрос: "Расскажите о сложном баге, который вы нашли"

S (Ситуация): "В проекте электронной коммерции..."
T (Задача): "Нужно было найти причину потери заказов..."
A (Действие): "Я проанализировал логи, написал SQL-запрос..."
R (Результат): "Нашёл race condition, который терял 2% заказов.
   Исправление сэкономило ~$50K/месяц."
```

- [ ] Какие ваши сильные стороны как Senior QA?

<details>
<summary>Ответ</summary>

Подготовьте 3-5 конкретных примеров: (1) Стратегическое мышление — пример построения тест-стратегии. (2) Техническая глубина — пример сложного бага. (3) Автоматизация — пример фреймворка или оптимизации. (4) Коммуникация — пример работы со стейкхолдерами. (5) Менторство — пример развития junior-коллеги.

- [ ] Как отвечать на вопрос "Какие ваши слабые стороны?"

<details>
<summary>Ответ</summary>

Выберите реальную область роста и покажите, что вы над ней работаете. Пример: "Раньше я недостаточно внимания уделяла нефункциональному тестированию. Сейчас я прохожу курс по нагрузочному тестированию и уже применила k6 для тестирования API в последнем проекте." Никогда не говорите "У меня нет слабостей" или "Я перфекционист".

</details>

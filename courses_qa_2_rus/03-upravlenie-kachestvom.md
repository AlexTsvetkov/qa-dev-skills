## 1. Метрики качества и KPI

```
  Ключевые метрики для QA лидера:

  ┌─────────────────────────┬────────────────────────────┐
  │ Метрика                 │ Формула / Цель             │
  ├─────────────────────────┼────────────────────────────┤
  │ Покрытие тестами        │ Покрыто / Всего × 100%    │
  │                         │ Цель: ≥ 95%               │
  ├─────────────────────────┼────────────────────────────┤
  │ Процент прохождения     │ Пройдено / Выполнено ×100%│
  │                         │ Цель: ≥ 95%               │
  ├─────────────────────────┼────────────────────────────┤
  │ Плотность дефектов      │ Баги / Модуль или Фича    │
  │                         │ Тренд: снижение            │
  ├─────────────────────────┼────────────────────────────┤
  │ Утечка дефектов         │ Прод баги / Все баги ×100% │
  │                         │ Цель: ≤ 10%               │
  ├─────────────────────────┼────────────────────────────┤
  │ Эффективность тестов    │ QA баги / (QA+Прод) × 100%│
  │                         │ Цель: ≥ 90%               │
  ├─────────────────────────┼────────────────────────────┤
  │ Среднее время           │ Дата закрытия - Дата       │
  │ исправления бага        │ открытия. Тренд: снижение │
  └─────────────────────────┴────────────────────────────┘

  Тренды дефектов (пример по спринтам):
    Спринт 1: Найдено 20, Исправлено 15  ← растёт
    Спринт 2: Найдено 18, Исправлено 20  ← стабилизация
    Спринт 3: Найдено 10, Исправлено 12  ← улучшение ✓
```

## 2. Тестирование на основе рисков

```
  Приоритизация тестирования по рискам:

  Риск = Вероятность × Влияние

  Матрица рисков:
  ┌──────────┬────────────┬────────────┬────────────┐
  │          │ Низкое     │ Среднее    │ Высокое    │
  │          │ влияние    │ влияние    │ влияние    │
  ├──────────┼────────────┼────────────┼────────────┤
  │ Высокая  │ СРЕДНИЙ    │ ВЫСОКИЙ    │ КРИТИЧНЫЙ  │
  │ вероятн. │            │            │            │
  ├──────────┼────────────┼────────────┼────────────┤
  │ Средняя  │ НИЗКИЙ     │ СРЕДНИЙ    │ ВЫСОКИЙ    │
  │ вероятн. │            │            │            │
  ├──────────┼────────────┼────────────┼────────────┤
  │ Низкая   │ НИЗКИЙ     │ НИЗКИЙ     │ СРЕДНИЙ    │
  │ вероятн. │            │            │            │
  └──────────┴────────────┴────────────┴────────────┘

  Распределение тестовых усилий:
    КРИТИЧНЫЙ → Полное покрытие + автоматизация
    ВЫСОКИЙ   → Полное покрытие
    СРЕДНИЙ   → Основные сценарии
    НИЗКИЙ    → Базовые проверки / smoke-тесты

  Критерии выхода (пример):
    ✓ 100% критичных тест-кейсов пройдено
    ✓ 95% всех тест-кейсов пройдено
    ✓ 0 открытых критичных/блокирующих багов
    ✓ ≤ 3 открытых багов среднего приоритета
    ✓ Все регрессионные тесты зелёные
    ✓ Покрытие требований ≥ 95%
```

## 5. Нефункциональное тестирование — обзор

```
  Типы нефункционального тестирования:

  Производительность:
    • Нагрузочное тестирование — поведение под нагрузкой
    • Стресс-тестирование — поведение за пределами нормы
    • Инструменты: JMeter, k6, Gatling

  Безопасность (OWASP Top 10):
    • Инъекции (SQL, XSS)
    • Нарушенная аутентификация
    • Утечка данных
    • Инструменты: OWASP ZAP, Burp Suite

  Кроссплатформенность:
    • Браузеры: Chrome, Firefox, Safari, Edge
    • Устройства: Desktop, Mobile, Tablet
    • ОС: Windows, macOS, Linux, iOS, Android

  Мобильное тестирование:
    • Специфика: разные размеры экранов
    • Сеть: offline, медленный интернет
    • Инструменты: BrowserStack, Appium
```

## ✅ Вопросы для самопроверки

1. Какие 5 ключевых метрик качества должен отслеживать QA лидер?

<details>
<summary>Ответ</summary>

(1) **Покрытие тестами** — % покрытых требований (цель ≥ 95%). (2) **Процент прохождения тестов** — пройдено/выполнено (цель ≥ 95%). (3) **Плотность дефектов** — баги на модуль/фичу (тренд на снижение). (4) **Утечка дефектов** — % багов, дошедших до продакшена (цель ≤ 10%). (5) **Эффективность тестирования** — % багов, найденных QA до продакшена (цель ≥ 90%).

</details>

2. Объясните тестирование на основе рисков и приведите пример критериев выхода.

<details>
<summary>Ответ</summary>

**Тестирование на основе рисков** приоритизирует усилия по формуле: Риск = Вероятность × Влияние. Критичные области получают максимальное покрытие, а низкорисковые — минимальное. Пример критериев выхода: 100% критичных тестов пройдено, 95% всех тестов пройдено, 0 критичных/блокирующих багов, ≤ 3 средних багов, покрытие требований ≥ 95%.

</details>

3. Как правильно коммуницировать риски качества стейкхолдерам?

<details>
<summary>Ответ</summary>

Используйте данные, а не мнения: конкретные метрики, проценты, числа. Всегда предлагайте решения, а не только проблемы (mitigation plan). Оценивайте влияние на бизнес (% пользователей, потери). Давайте чёткие рекомендации: Go/No-Go, условный релиз с hotfix. Используйте цветовую индикацию (🟢🟡🔴) для быстрого понимания.

</details>

4. Какие обязанности у QA лидера в области менторинга?

<details>
<summary>Ответ</summary>

QA лидер отвечает за: (1) парное тестирование — совместные сессии с младшими QA, (2) обзор тест-кейсов с конструктивной обратной связью, (3) делегирование задач с поддержкой и контролем, (4) регулярные 1:1 встречи для обсуждения развития, (5) обмен знаниями через внутренние презентации и документацию.

</details>

5. Перечислите основные направления нефункционального тестирования.

<details>
<summary>Ответ</summary>

(1) **Производительность** — нагрузочное и стресс-тестирование (JMeter, k6). (2) **Безопасность** — проверка OWASP Top 10: инъекции, аутентификация, утечки данных (OWASP ZAP). (3) **Кроссплатформенность** — разные браузеры, ОС, устройства. (4) **Мобильное тестирование** — разные экраны, сетевые условия, offline-режим.

</details>

## 1. Классы эквивалентности

```
  Разделение входных данных на группы (классы),
  где поведение системы ОДИНАКОВО внутри класса.

  Пример: Поле "Возраст" (допустимо: 18–65)

  ┌──────────────────┬───────────────┬──────────┐
  │ Класс            │ Значения      │ Тип      │
  ├──────────────────┼───────────────┼──────────┤
  │ Невалидный (ниже)│ …, 16, 17    │ Негативн.│
  │ Валидный         │ 18, 19…64, 65│ Позитивн.│
  │ Невалидный (выше)│ 66, 67, …    │ Негативн.│
  └──────────────────┴───────────────┴──────────┘

  Правило: Достаточно 1 значения из каждого класса:
    Невалидный ниже: 15
    Валидный: 30
    Невалидный выше: 70
```

## 2. Анализ граничных значений

```
  Баги чаще всего на ГРАНИЦАХ диапазонов.

  Поле "Возраст" (допустимо: 18–65):

  Невалидно │ Валидно          │ Невалидно
  ──────────┼──────────────────┼──────────
       17   │ 18  19 ... 64 65│ 66
            │ ↑              ↑ │
            │ Нижняя    Верхняя│
            │ граница   граница│

  Граничные значения для тестов:
  ┌──────────┬──────────┬──────────────────┐
  │ Значение │ Ожидание │ Тип              │
  ├──────────┼──────────┼──────────────────┤
  │ 17       │ Ошибка   │ Ниже нижней      │
  │ 18       │ Успех    │ Нижняя граница   │
  │ 19       │ Успех    │ Выше нижней      │
  │ 64       │ Успех    │ Ниже верхней     │
  │ 65       │ Успех    │ Верхняя граница  │
  │ 66       │ Ошибка   │ Выше верхней     │
  └──────────┴──────────┴──────────────────┘
```

## 3. Таблицы решений

```
  Когда результат зависит от КОМБИНАЦИИ условий.

  Пример: Скидка в магазине
    Условие 1: Член клуба? (Да/Нет)
    Условие 2: Покупка > 5000 руб? (Да/Нет)
    Условие 3: Праздничный день? (Да/Нет)

  ┌─────────────┬────┬────┬────┬────┬────┬────┬────┬────┐
  │             │ R1 │ R2 │ R3 │ R4 │ R5 │ R6 │ R7 │ R8 │
  ├─────────────┼────┼────┼────┼────┼────┼────┼────┼────┤
  │ Член клуба  │ Да │ Да │ Да │ Да │Нет │Нет │Нет │Нет │
  │ > 5000 руб  │ Да │ Да │Нет │Нет │ Да │ Да │Нет │Нет │
  │ Праздник    │ Да │Нет │ Да │Нет │ Да │Нет │ Да │Нет │
  ├─────────────┼────┼────┼────┼────┼────┼────┼────┼────┤
  │ Скидка      │30% │20% │15% │10% │15% │10% │ 5% │ 0% │
  └─────────────┴────┴────┴────┴────┴────┴────┴────┴────┘

  Каждое правило (R1–R8) = один тест-кейс.
```

## 4. Диаграммы переходов состояний

```
  Для систем с разными состояниями и переходами.

  Пример: Заказ в интернет-магазине

  [Новый] ──создан──→ [Подтверждён]
                           │
                      оплачен
                           │
                           ▼
                     [Оплачен] ──отправлен──→ [В доставке]
                           │                       │
                      отменён               доставлен
                           │                       │
                           ▼                       ▼
                    [Отменён]             [Доставлен]

  Тест-кейсы из диаграммы:
    1. Новый → Подтверждён → Оплачен → В доставке → Доставлен ✓
    2. Новый → Подтверждён → Оплачен → Отменён ✓
    3. Новый → Подтверждён → отмена? (зависит от бизнес-правил)
    4. Доставлен → Оплачен? (невозможный переход → тест!) ✓
    
  Негативные тесты: попытки невозможных переходов.
```

## 5. Исследовательское тестирование

```
  Структурированное исследование без заранее

  ┌──────────────────────────────────────────────┐
  │ Сессия: 60-90 минут                          │
  │                                              │
  │ Хартия (Charter):                            │

  │                                              │
  │ Записываем:                                  │
  │ • Что тестировали                            │
  │ • Найденные баги                             │
  │ • Заметки и вопросы                          │
  │ • Области для дальнейшего исследования       │
  │                                              │
  │ Результат:                                   │
  │ • Время: 75 минут                            │

  └──────────────────────────────────────────────┘

  Когда использовать:
    • Новая фича без документации
    • Дополнение к формальным тестам
    • Ограниченное время
    • Поиск неочевидных багов
```

## 6. SQL для валидации данных

```
  Основные запросы для QA:

  -- Выборка данных
  SELECT * FROM orders WHERE status = 'active';

  -- Поиск дубликатов
  SELECT email, COUNT(*) as cnt
  FROM users
  GROUP BY email
  HAVING COUNT(*) > 1;

  -- Проверка целостности данных
  SELECT o.id, o.user_id
  FROM orders o
  LEFT JOIN users u ON o.user_id = u.id
  WHERE u.id IS NULL;  -- Заказы без пользователя!

  -- Агрегация для отчётов
  SELECT status, COUNT(*) as total
  FROM orders
  GROUP BY status;

  -- Кросс-системная валидация
  SELECT a.order_id, a.amount, b.amount
  FROM system_a.payments a
  JOIN system_b.payments b ON a.order_id = b.order_id
  WHERE a.amount != b.amount;  -- Расхождения!
```

## 7. Подготовка к собеседованию

```
  Ключевые темы для Senior QA:

  1. Стратегия и лидерство:
     • Как вы строите тест-стратегию?
     • Как приоритизируете тестирование?
     • Как принимаете Go/No-Go решения?

  2. Автоматизация:
     • Какие фреймворки использовали?
     • Как интегрируете тесты в CI/CD?
     • Как боретесь с flaky тестами?

  3. Технический QA:
     • API тестирование — подходы и инструменты
     • SQL — сложные запросы для валидации
     • Тест-дизайн — какие техники и когда

  4. AI-тестирование:
     • Как тестировать AI/LLM системы?
     • Какие метрики для AI?
     • Как использовать GenAI в работе QA?

  5. Коммуникация:
     • Как объясняете риски стейкхолдерам?
     • Как решаете конфликты приоритетов?
     • Как менторите младших QA?

  STAR-метод для ответов:
    S — Ситуация (контекст)
    T — Задача (что нужно было сделать)
    A — Действие (что вы сделали)
    R — Результат (чего достигли)
```

## ✅ Вопросы для самопроверки

1. Объясните разницу между классами эквивалентности и граничными значениями.

<details>
<summary>Ответ</summary>

**Классы эквивалентности** — разделение входных данных на группы с одинаковым поведением системы. Достаточно 1 значения из класса. **Граничные значения** — тестирование значений на границах классов, где баги наиболее вероятны. Для диапазона 18–65: классы эквивалентности дают 3 теста (15, 30, 70), а граничные значения — 6 тестов (17, 18, 19, 64, 65, 66).

</details>

2. Когда использовать таблицы решений?

<details>
<summary>Ответ</summary>

Когда результат зависит от **комбинации нескольких условий**. Каждая комбинация условий (правило) = отдельный тест-кейс. Пример: скидка зависит от членства, суммы покупки и праздничного дня. С 3 булевыми условиями получается 2³ = 8 правил = 8 тест-кейсов, покрывающих все комбинации.

</details>

3. Как проводить сессионное исследовательское тестирование?

<details>
<summary>Ответ</summary>

(1) Определить **хартию** — цель сессии (что и зачем исследуем). (2) Ограничить время — 60-90 минут. (3) Во время сессии записывать: что тестировали, найденные баги, заметки, вопросы. (4) По итогам — отчёт: время, найденные баги, покрытие, области для дальнейшего исследования. Используется для новых фич без документации, дополнение к формальным тестам.

</details>

4. Напишите SQL для поиска дубликатов в таблице.

<details>
<summary>Ответ</summary>

```sql
SELECT email, COUNT(*) as cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

`GROUP BY` группирует записи по полю, `COUNT(*)` считает количество в каждой группе, `HAVING COUNT(*) > 1` фильтрует только группы с дубликатами (более 1 записи).

</details>

5. Что такое STAR-метод и как его применять на собеседовании?

<details>
<summary>Ответ</summary>

**STAR** — структура для ответов на поведенческие вопросы: **S (Situation)** — описание контекста и ситуации, **T (Task)** — какая задача стояла, **A (Action)** — какие конкретные действия вы предприняли, **R (Result)** — какой результат получили (желательно с цифрами). Пример: «В проекте X (S) нужно было снизить утечку багов (T), я внедрил регрессионную автоматизацию (A), утечка снизилась с 15% до 3% (R)».

</details>

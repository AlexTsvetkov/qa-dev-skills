## 1. Agile и Scrum для QA

```
  Scrum-спринт (2 недели):

  Планирование → Ежедневные стендапы → Обзор → Ретро
  ┌──────────────────────────────────────────────────┐
  │ День 1        │ Дни 2-9        │ День 10        │
  │ Планирование  │ Работа спринта │ Обзор + Ретро  │
  └──────────────────────────────────────────────────┘

  Роль QA в Scrum:
  ┌──────────────────┬──────────────────────────────────┐
  │ Событие          │ Участие QA                       │
  ├──────────────────┼──────────────────────────────────┤
  │ Планирование     │ Оценка историй, критерии приёмки │
  │ Стендап          │ Статус тестирования, блокеры     │
  │ Груминг бэклога  │ Тестируемость, вопросы           │
  │ Обзор спринта    │ Демо тестовых результатов        │
  │ Ретроспектива    │ Улучшение процесса качества      │
  └──────────────────┴──────────────────────────────────┘

  Definition of Done (DoD) — с участием QA:
    ✓ Код написан и прошёл ревью
    ✓ Юнит-тесты написаны и прошли
    ✓ Функциональные тесты пройдены
    ✓ Регрессионные тесты зелёные
    ✓ Баги исправлены или задокументированы
    ✓ Документация обновлена
```

## 3. Shift-Left подход

```
  Традиционный подход:
    Требования → Дизайн → Код → ════════ ТЕСТИРОВАНИЕ ════════
                                  (поздно, дорого, рискованно)

  Shift-Left:
    ════ ТЕСТИРОВАНИЕ ════════════════════════════════════════
    Требования → Дизайн → Код → Интеграция → Релиз
    ↑            ↑         ↑       ↑            ↑
    Обзор        Тест-     Юнит-   API-         E2E
    требований   дизайн    тесты   тесты        тесты

  Принципы Shift-Left:
    1. Раннее вовлечение QA (с этапа требований)
    2. Код-ревью с точки зрения тестируемости
    3. Автоматизация на ранних этапах
    4. Быстрая обратная связь через CI/CD

  Стоимость исправления бага:
    Требования:  $1   (нашли рано = дёшево)
    Дизайн:      $5
    Код:         $10
    Тестирование: $50
    Продакшен:   $500  (нашли поздно = дорого!)
```

## 4. DevOps и непрерывное тестирование

```
  DevOps Pipeline:
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ Plan │→ │ Code │→ │Build │→ │ Test │→ │Deploy│→ │Monit.│
  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
                          ↓         ↓         ↓         ↓
                       Компил.   Авто-     Staging   Логи,
                       Lint      тесты    Canary    Алерты

  Непрерывное тестирование = тесты на КАЖДОМ этапе:
    Commit  → Линтинг + юнит-тесты (секунды)
    Build   → Интеграционные тесты (минуты)
    Staging → E2E + smoke-тесты (минуты)
    Prod    → Мониторинг + health checks (постоянно)

  Типичный CI/CD Pipeline:
    1. Разработчик пушит код
    2. CI сервер запускает сборку
    3. Юнит-тесты (быстрые, < 5 мин)
    4. Интеграционные тесты (< 15 мин)
    5. Деплой на staging
    6. E2E тесты (< 30 мин)
    7. Ручная проверка (если нужно)
    8. Деплой на продакшен
    9. Smoke-тесты на проде
```

## 5. CI/CD инструменты для QA

```
  ┌──────────────────┬──────────────────────────────────┐
  │ Инструмент       │ Использование QA                 │
  ├──────────────────┼──────────────────────────────────┤
  │ Jenkins          │ Настройка тестовых джобов        │
  │ GitHub Actions   │ Тесты в workflow'ах              │
  │ GitLab CI/CD     │ Pipeline с тестовыми стадиями    │
  │ Azure DevOps     │ Тестовые планы + пайплайны       │
  └──────────────────┴──────────────────────────────────┘

  Пример GitHub Actions для тестов:

  name: Test Pipeline
  on: [push, pull_request]
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
        - name: Run unit tests
          run: npm test
        - name: Run API tests
          run: npm run test:api
        - name: Run E2E tests
          run: npm run test:e2e
```

## ✅ Вопросы для самопроверки

1. Как QA участвует в каждом событии Scrum?

<details>
<summary>Ответ</summary>

**Планирование** — оценка историй с точки зрения тестирования, уточнение критериев приёмки. **Ежедневный стендап** — статус тестирования, блокеры. **Груминг бэклога** — вопросы по тестируемости, выявление рисков. **Обзор спринта** — демонстрация тестовых результатов, статус качества. **Ретроспектива** — предложения по улучшению процесса качества, анализ пропущенных багов.

</details>

2. Что такое Shift-Left и почему это важно?

<details>
<summary>Ответ</summary>

**Shift-Left** — подход, при котором тестирование начинается как можно раньше в жизненном цикле разработки: обзор требований, тест-дизайн на этапе проектирования, юнит-тесты при написании кода. Это важно, потому что стоимость исправления бага растёт экспоненциально: баг в требованиях стоит $1, а в продакшене — $500. Раннее обнаружение = дешевле + меньше рисков.

</details>

3. Опишите типичный CI/CD пайплайн с интеграцией тестов.

<details>
<summary>Ответ</summary>

(1) Разработчик пушит код → (2) CI сервер запускает сборку → (3) **Юнит-тесты** (быстрые, < 5 мин) → (4) **Интеграционные тесты** (< 15 мин) → (5) Деплой на staging → (6) **E2E тесты** (< 30 мин) → (7) Ручная проверка (если нужна) → (8) Деплой на продакшен → (9) **Smoke-тесты** на проде. Каждый этап — гейт: если тесты упали, пайплайн останавливается.

</details>

4. Что такое Definition of Done и как QA влияет на него?

<details>
<summary>Ответ</summary>

**Definition of Done (DoD)** — набор критериев, которым должна соответствовать история, чтобы считаться завершённой. QA вносит в DoD: функциональные тесты пройдены, регрессионные тесты зелёные, баги исправлены или задокументированы, автоматические тесты обновлены, тестовые данные очищены. Без выполнения всех критериев DoD история не считается «Done».

</details>

5. Чем DevOps-культура отличается от традиционного подхода для QA?

<details>
<summary>Ответ</summary>

**Традиционный подход:** QA — отдельная фаза после разработки, ручное тестирование, редкие релизы, тестирование «в конце». **DevOps-культура:** QA интегрировано в каждый этап, автоматизация тестов в CI/CD пайплайне, непрерывное тестирование, быстрая обратная связь, мониторинг в продакшене, частые релизы. QA — не gate keeper, а enabler качества.

</details>

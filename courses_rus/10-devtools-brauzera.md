# Курс 10: DevTools браузера для QA

## 📖 Что вы изучите

- Как открыть и ориентироваться в DevTools браузера
- Использование вкладки Network для проверки API-вызовов
- Использование вкладки Console для поиска JavaScript-ошибок
- Использование вкладки Elements для инспекции HTML/CSS
- Использование вкладки Application для проверки хранилищ (cookies, localStorage)
- Практические сценарии отладки для QA

---

## 1. Открытие DevTools

| Способ | Сочетание клавиш |
|--------|------------------|
| **Клавиатура (Mac)** | `Cmd + Option + I` |
| **Клавиатура (Windows/Linux)** | `F12` или `Ctrl + Shift + I` |
| **Правый клик** | Правый клик по любому элементу → «Просмотреть код» |
| **Меню** | Chrome → Дополнительные инструменты → Инструменты разработчика |

### Обзор вкладок DevTools:

| Вкладка | Что делает | Применение в QA |
|---------|-----------|----------------|
| **Elements** | Инспекция HTML/CSS | Проверка структуры UI, стилей |
| **Console** | JavaScript-логи и ошибки | Поиск JS-ошибок |
| **Network** | Все HTTP-запросы/ответы | Проверка API-вызовов ⭐ |
| **Application** | Хранилища, cookies, кэш | Проверка токенов авторизации, данных |
| **Performance** | Производительность загрузки страницы | Поиск медленных страниц |
| **Sources** | Исходный код JavaScript | Отладка скриптов |

## 2. Вкладка Network (самая важная для QA)

Вкладка Network показывает **каждый HTTP-запрос**, который делает страница.

### Как использовать:

1. Откройте DevTools → вкладка **Network**
2. **Перезагрузите страницу** (запросы появляются только после открытия вкладки)
3. Взаимодействуйте с приложением — каждый API-вызов появляется в списке

### Чтение сетевого запроса:

```
┌─────────────────────────────────────────────────────────────┐
│ Network                                    🔴 Запись        │
├──────┬──────────┬────────┬──────┬──────────┬───────────────┤
│Статус│ Имя      │ Метод  │ Тип  │ Размер   │ Время         │
├──────┼──────────┼────────┼──────┼──────────┼───────────────┤
│ 200  │ /api/user│ GET    │ json │ 1.2 KB   │ 45 ms         │
│ 201  │ /api/task│ POST   │ json │ 0.5 KB   │ 120 ms        │
│ 404  │ /api/old │ GET    │ json │ 0.1 KB   │ 23 ms    ❌   │
│ 500  │ /api/save│ POST   │ json │ 0.2 KB   │ 5000 ms  ❌   │
└──────┴──────────┴────────┴──────┴──────────┴───────────────┘
```

### Инспекция отдельного запроса:

Кликните по любому запросу, чтобы увидеть подробности:

**Вкладка Headers:**
```
Request URL: https://api.example.com/api/users/42
Request Method: GET
Status Code: 200 OK

Request Headers:
  Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
  Content-Type: application/json
  Accept: application/json

Response Headers:
  Content-Type: application/json
  X-Request-Id: abc-123-def
```

**Вкладка Response (Preview):**
```json
{
  "id": 42,
  "name": "Nastya",
  "email": "nastya@example.com",
  "role": "qa"
}
```

**Вкладка Payload (для POST/PUT запросов):**
```json
{
  "name": "Nastya",
  "email": "nastya@example.com"
}
```

### Фильтры вкладки Network:

| Фильтр | Показывает |
|--------|-----------|
| **All** | Всё |
| **Fetch/XHR** | Только API-вызовы (самый полезный для QA ⭐) |
| **JS** | JavaScript-файлы |
| **CSS** | Таблицы стилей |
| **Img** | Изображения |
| **Doc** | HTML-документы |

### Советы QA по вкладке Network:

- 🔴 **Красные запросы** = ошибки (коды статуса 4xx, 5xx)
- 🕐 **Медленные запросы** = проблемы с производительностью (столбец Time)
- 🔍 **Фильтруйте по «Fetch/XHR»**, чтобы видеть только API-вызовы
- 📋 **Правый клик → Copy as cURL** для воспроизведения запроса в терминале
- 🚫 **Проверяйте неудачные запросы**, которые UI может молча игнорировать

## 3. Вкладка Console

Показывает **JavaScript-логи, предупреждения и ошибки**.

### Типичные сообщения:

```
// Ошибка (красная) — что-то сломалось
❌ Uncaught TypeError: Cannot read property 'name' of undefined
   at UserProfile.js:42

// Предупреждение (жёлтое) — потенциальная проблема
⚠️ React Warning: Each child in a list should have a unique "key" prop

// Информация (синяя) — информационное сообщение
ℹ️ API response received in 234ms

// Лог (белый) — отладочный вывод
📝 User logged in: nastya@example.com
```

### Советы QA по вкладке Console:

- **Красные ошибки** = JavaScript-баги — фиксируйте их в баг-репорте!
- **Обращайте внимание на файл и номер строки** (например, `UserProfile.js:42`)
- **Проверяйте консоль после воспроизведения бага** — ошибки часто появляются здесь
- **CORS-ошибки** появляются здесь, когда API-вызовы заблокированы:
  ```
  ❌ Access to fetch at 'https://api.example.com' from origin
     'https://app.example.com' has been blocked by CORS policy
  ```

## 4. Вкладка Elements

Инспекция и изменение **HTML-структуры и CSS-стилей** в реальном времени.

### Как использовать:

1. **Правый клик по любому элементу** → «Просмотреть код»
2. Вкладка Elements подсветит HTML-элемент
3. Правая панель покажет применённые CSS-стили

### Что может делать QA:

| Действие | Как | Зачем |
|----------|-----|-------|
| **Проверить текст элемента** | Кликнуть по элементу в дереве HTML | Проверить отображаемый текст |
| **Проверить CSS** | Посмотреть панель Styles | Проверить цвета, шрифты, отступы |
| **Редактировать текст вживую** | Двойной клик по тексту в HTML | Тестировать длинный текст, граничные случаи |
| **Скрыть элемент** | Нажать `H` на выделенном элементе | Тестировать вёрстку без элемента |
| **Проверить адаптивность** | Кликнуть 📱 панель устройств | Тестировать мобильные макеты |

### Панель устройств (тестирование адаптивности):

1. Кликните по **иконке устройства** 📱 (или `Cmd+Shift+M` / `Ctrl+Shift+M`)
2. Выберите пресет устройства (iPhone, iPad и др.) или задайте свои размеры
3. Проверьте, как страница выглядит на разных размерах экрана

## 5. Вкладка Application

Инспекция **cookies, localStorage, sessionStorage и кэша**.

### Cookies:

```
┌──────────────────────────────────────────────────────┐
│ Cookies > https://example.com                         │
├──────────────┬──────────────────┬─────────┬──────────┤
│ Имя          │ Значение         │ Срок    │ HttpOnly │
├──────────────┼──────────────────┼─────────┼──────────┤
│ session_id   │ abc123def456     │ Сессия  │ ✅       │
│ auth_token   │ eyJhbGciOi...    │ 1 час   │ ✅       │
│ user_pref    │ dark_mode=true   │ 1 год   │ ❌       │
│ _ga          │ GA1.2.123456     │ 2 года  │ ❌       │
└──────────────┴──────────────────┴─────────┴──────────┘
```

### localStorage:

```
┌──────────────────────────────────────────┐
│ Local Storage > https://example.com       │
├──────────────────┬───────────────────────┤
│ Ключ             │ Значение              │
├──────────────────┼───────────────────────┤
│ user_settings    │ {"theme":"dark"}      │
│ recent_searches  │ ["api","test","bug"]  │
│ cart_items       │ [{"id":1,"qty":2}]    │
└──────────────────┴───────────────────────┘
```

### Советы QA по вкладке Application:

- 🍪 **Удалите cookies** для тестирования выхода / истечения сессии
- 🗑️ **Очистите localStorage** для тестирования пользовательского опыта с нуля
- 🔑 **Проверьте токены авторизации** — присутствуют ли они? Не истёк ли их срок?
- 📦 **Проверьте кэшированные данные** — не вызывают ли устаревшие данные проблемы?

## 6. Сценарии отладки для QA

### Сценарий 1: «Кнопка не работает»

```
1. Откройте DevTools → вкладка Console
2. Нажмите сломанную кнопку
3. Проверьте наличие JavaScript-ошибок (красные сообщения)
4. Переключитесь на вкладку Network → нажмите кнопку снова
5. Проверьте, был ли отправлен API-запрос
6. Если да: проверьте статус ответа и тело
7. Если нет: это проблема фронтенд-JavaScript
```

### Сценарий 2: «Страница показывает неправильные данные»

```
1. Откройте DevTools → вкладка Network → фильтр «Fetch/XHR»
2. Перезагрузите страницу
3. Найдите API-вызов, загружающий данные
4. Проверьте тело ответа — корректны ли данные от API?
5. Если данные от API корректны → баг отображения на фронтенде
6. Если данные от API некорректны → баг на бэкенде
```

### Сценарий 3: «Страница медленно загружается»

```
1. Откройте DevTools → вкладка Network
2. Перезагрузите страницу
3. Посмотрите нижнюю строку: «X requests, X MB transferred, Finish: X s»
4. Отсортируйте по столбцу Time — найдите самые медленные запросы
5. Проверьте, занимает ли какой-нибудь запрос > 2 секунд
6. Проверьте наличие больших ответов (столбец Size)
7. Проверьте количество запросов (не слишком ли их много)
```

### Сценарий 4: «Не могу войти в систему»

```
1. Откройте DevTools → вкладка Network
2. Попробуйте выполнить вход
3. Найдите API-запрос логина (POST /api/auth/login)
4. Проверьте тело запроса — правильно ли отправлены учётные данные?
5. Проверьте ответ — какую ошибку возвращает API?
6. Проверьте вкладку Application → Cookies — установлена ли cookie сессии?
7. Проверьте Console — нет ли CORS или ошибок безопасности?
```

## 📝 Упражнения

### Упражнение 1: Изучите вкладку Network

На любом веб-сайте:
1. Откройте DevTools → вкладка Network → фильтр «Fetch/XHR»
2. Походите по сайту
3. Найдите как минимум 3 API-вызова и для каждого запишите:
   - URL, метод, код статуса
   - Содержимое тела ответа

### Упражнение 2: Найдите информацию

Используя DevTools, найдите:
1. Все cookies, установленные сайтом
2. Любые JavaScript-ошибки в Console
3. CSS-цвет основного заголовка
4. Является ли сайт адаптивным (проверьте через панель устройств)

### Упражнение 3: Отладьте сценарий

Опишите шаги в DevTools, которые вы бы предприняли для отладки:
1. Отправка формы показывает «Ошибка» без подробностей
2. Изображение не загружается
3. Страница загружается 10 секунд
4. Пользователя неожиданно выкидывает из системы

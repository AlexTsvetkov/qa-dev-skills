## 1. Какую проблему решает CI/CD

### Без CI/CD (старый подход):

```
Разработчик пишет код
        ↓
Ждёт недели для слияния с кодом других разработчиков
        ↓
"День интеграции" — все сливают код → хаос, конфликты, поломки
        ↓
Ручное тестирование в течение дней/недель
        ↓
Ручной деплой (SSH на сервер, копировать файлы, молиться)
        ↓
Что-то ломается в продакшене в 3 часа ночи
        ↓
😱 Паника
```

### С CI/CD:

```
Разработчик пушит код
        ↓
Автоматически собирается, тестируется и проверяется (за минуты)
        ↓
Автоматически деплоится в тестовое окружение
        ↓
QA тестирует на окружении
        ↓
Одобрено → автоматический деплой в продакшен
        ↓
😎 Уверенность
```

## 2. Что такое CI/CD?

CI/CD — это три связанные практики:

### CI — Continuous Integration (Непрерывная интеграция)

**«Сливайте код часто. Собирайте и тестируйте автоматически.»**

```
Разработчик A пушит код ──→ ┌──────────────┐
                            │   CI-сервер   │ → Сборка → Тесты → ✅ или ❌
Разработчик B пушит код ──→ │ (автоматика)  │
                            └──────────────┘
```

**Что происходит:**
1. Разработчики пушат код в общий репозиторий (Git) несколько раз в день
2. Каждый пуш запускает **автоматическую сборку**
3. **Автоматические тесты** запускаются для проверки сборки
4. Если что-то падает → команда немедленно уведомляется
5. Цель: код всегда в **собираемом, тестируемом состоянии**

**Преимущества:**
- 🐛 Баги находятся быстро (через минуты после появления)
- 🔀 Конфликты слияния маленькие и легко разрешаются
- ✅ Каждый коммит проверяется автотестами
- 👥 Команда уверена, что код работает

### CD — Continuous Delivery (Непрерывная доставка)

**«Код всегда готов к деплою. Деплой — это решение в один клик.»**

```
CI прошёл ──→ Деплой в Staging ──→ QA-тестирование ──→ [Ручное одобрение] ──→ Деплой в Продакшен
                                                        ↑
                                                  Решает человек
```

**Что происходит:**
1. После прохождения CI код **автоматически деплоится в staging-окружение**
2. QA и стейкхолдеры тестируют на staging
3. После одобрения **один клик** деплоит в продакшен
4. Код всегда в **готовом к деплою состоянии**

### CD — Continuous Deployment (Непрерывный деплой)

**«Каждое изменение, прошедшее все тесты, автоматически идёт в продакшен.»**

```
CI прошёл ──→ Деплой в Staging ──→ Автотесты прошли ──→ Деплой в Продакшен (автоматически!)
```

**Разница с Continuous Delivery:**
- **Delivery** = автоматический деплой в staging, **ручное** одобрение для продакшена
- **Deployment** = автоматический деплой **везде**, включая продакшен

Большинство компаний используют **Continuous Delivery** (с ручным одобрением для продакшена).

## 3. CI/CD-пайплайн

**Пайплайн** — это автоматизированный рабочий процесс, через который проходит код от коммита до деплоя.

### Типичные стадии пайплайна:

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  ИСХОДНИК │───→│  СБОРКА  │───→│  ТЕСТЫ   │───→│  ДЕПЛОЙ  │───→│  ДЕПЛОЙ  │
│           │    │          │    │          │    │ (staging) │    │  (prod)  │
│ Git push  │    │ Компилир.│    │Запуск    │    │          │    │          │
│ Merge PR  │    │ Упаковка │    │тестов    │    │ Авто     │    │ Ручное   │
│           │    │          │    │Анализ    │    │          │    │одобрение │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     ✅              ✅              ✅              ✅              ✅
  Запуск         Компиляция    Unit-тесты     Деплой в       Деплой в
  пайплайна      кода          Интеграционные staging        продакшен
                               тесты
                               Качество кода
```

### Детали стадий:

#### Стадия 1: Исходник (Триггер)
- Пайплайн запускается при пуше кода или создании Pull Request
- Может также запускаться по расписанию (например, ночные сборки)

#### Стадия 2: Сборка
- **Компиляция** исходного кода
- **Загрузка зависимостей** (библиотек, необходимых коду)
- **Упаковка** приложения (создание JAR-файла, Docker-образа и т.д.)
- Если код не компилируется → пайплайн падает ❌

#### Стадия 3: Тесты
- **Unit-тесты** — тестирование отдельных функций/методов
- **Интеграционные тесты** — тестирование компонентов вместе
- **Статический анализ** — проверка качества кода без его запуска (SonarQube, линтеры)
- **Проверки безопасности** — поиск известных уязвимостей
- Если хоть один тест падает → пайплайн падает ❌

#### Стадия 4: Деплой в Staging
- Деплой собранного приложения в **staging-окружение**
- Staging — копия продакшена (та же конфигурация, похожие данные)
- QA и стейкхолдеры могут тестировать здесь

#### Стадия 5: Деплой в Продакшен
- После одобрения — деплой в **продакшен-окружение**
- Это то, что видят реальные пользователи
- Обычно требуется **ручное одобрение** или дополнительные проверки

## 4. Ключевые концепции

### Пайплайн (Pipeline)
Весь автоматизированный рабочий процесс от начала до конца.

### Стадия (Stage)
Группа связанных шагов (например, «Сборка», «Тесты», «Деплой»). Стадии обычно выполняются последовательно.

### Задача (Job)
Конкретная задача внутри стадии. Задачи внутри одной стадии могут выполняться **параллельно**.

```
Пайплайн
├── Стадия: Сборка
│   └── Задача: Компиляция и упаковка
├── Стадия: Тесты
│   ├── Задача: Unit-тесты              (параллельно)
│   ├── Задача: Интеграционные тесты    (параллельно)
│   └── Задача: Проверка качества кода  (параллельно)
├── Стадия: Деплой в Staging
│   └── Задача: Деплой на staging-сервер
└── Стадия: Деплой в Продакшен
    └── Задача: Деплой на production-сервер
```

### Артефакт (Artifact)
Файл, созданный пайплайном (JAR-файл, Docker-образ, отчёт о тестах). Артефакты передаются между стадиями.

### Окружение (Environment)
Где работает приложение:

| Окружение | Назначение | Кто использует |
|-----------|-----------|---------------|
| **Development (Dev)** | Разработчики тестируют свои изменения | Разработчики |
| **Testing / QA** | QA-команда тестирует фичи | QA-команда |
| **Staging** | Предпродакшен-тестирование | QA, стейкхолдеры |
| **Production (Prod)** | Живая система для пользователей | Конечные пользователи |

### Триггер (Trigger)
Что запускает пайплайн:

| Триггер | Когда |
|---------|-------|
| **Push** | Код пушится в ветку |
| **Pull Request** | Создан или обновлён PR |
| **Merge** | Код слит в main/master |
| **Расписание** | Запуск в определённое время (cron) |
| **Ручной** | Кто-то нажимает «Запустить пайплайн» |

## 5. Контроль версий и ветвление (основы Git для CI/CD)

CI/CD тесно связан с Git. Вот как команды обычно используют ветки:

### Типичная стратегия ветвления:

```
main (продакшен)   ────●────────────────●──────────────────●────→
                       │                ↑                  ↑
                       │     merge PR   │       merge PR   │
                       │                │                  │
develop            ────●──●──●──●──●────●──●──●──●──●──●──●────→
                          │        ↑       │           ↑
                          │  merge │       │     merge │
                          │        │       │           │
feature/login         ────●──●──●──┘       │           │
                                           │           │
feature/user-api                       ────●──●──●──●──┘
```

| Ветка | Назначение | Поведение CI/CD |
|-------|-----------|----------------|
| `main` / `master` | Готовый к продакшену код | Деплоится в продакшен |
| `develop` | Последний код разработки | Деплоится в staging |
| `feature/*` | Разработка новых фич | Только запуск тестов |
| `release/*` | Подготовка релиза | Деплоится в staging |
| `hotfix/*` | Экстренные исправления | Быстрый путь в продакшен |

### Pull Requests (PRs):

Pull Request — это запрос на слияние кода из одной ветки в другую:

```
feature/login ──→ [Pull Request] ──→ develop
                        │
                        ├── CI-пайплайн запускается автоматически
                        ├── Код-ревью членами команды
                        ├── Все тесты должны пройти ✅
                        └── После одобрения → слияние
```

**Проверки CI/CD на PR:**
- ✅ Код компилируется
- ✅ Все тесты проходят
- ✅ Качество кода соответствует стандартам
- ✅ Нет уязвимостей безопасности
- ✅ Код-ревью одобрено

## 6. Инструменты CI/CD

| Инструмент | Тип | Используется |
|-----------|-----|-------------|
| **Jenkins** | Локальный CI/CD-сервер | Крупные предприятия |
| **GitHub Actions** | Облачный, встроен в GitHub | GitHub-проекты |
| **GitLab CI/CD** | Встроен в GitLab | GitLab-проекты |
| **Azure DevOps** | Облачный CI/CD от Microsoft | Enterprise, пользователи Azure |
| **CircleCI** | Облачный CI/CD | Стартапы, open source |
| **Travis CI** | Облачный CI/CD | Open source проекты |
| **TeamCity** | Локальный (JetBrains) | Экосистема JetBrains |

### Пример: GitHub Actions пайплайн

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Java
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Build with Maven
        run: mvn clean compile

  test:
    needs: build                    # Запускается после успешной сборки
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Java
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Run unit tests
        run: mvn test

      - name: Run integration tests
        run: mvn verify -P integration-tests

  deploy-staging:
    needs: test                     # Запускается после успешных тестов
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        run: echo "Deploying to staging..."

  deploy-production:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production          # Требует ручного одобрения
    steps:
      - name: Deploy to production
        run: echo "Deploying to production..."
```

## 7. Как CI/CD помогает QA

### До CI/CD:
- «У меня на машине работает» 🤷 — нет стандартного процесса сборки
- «Я что-то сломал?» — нет быстрой обратной связи
- «Когда будет готов билд?» — ручные сборки занимают время
- «Staging актуален?» — ручные деплои

### После CI/CD:

| Польза для QA | Как |
|--------------|-----|
| **Быстрая обратная связь** | За минуты узнаёте, сломало ли изменение что-то |
| **Единообразные окружения** | Один и тот же процесс сборки каждый раз |
| **Staging всегда готов** | Новый код автоматически деплоится в staging |
| **Отчёты о тестах** | Результаты автотестов доступны в пайплайне |
| **Уверенность в релизах** | Несколько стадий проверки перед продакшеном |
| **Возможность отката** | Можно быстро задеплоить предыдущую версию |

### Что QA должен знать о пайплайне:

1. **Где смотреть статус билда** — Последний билд проходит? ✅ или ❌
2. **Где смотреть отчёты о тестах** — Какие тесты прошли/упали?
3. **Как запустить билд** — Можно ли перезапустить пайплайн?
4. **Что задеплоено** — Какая версия на staging? На продакшене?
5. **Как читать логи пайплайна** — Если что-то упало, куда смотреть?

### Упражнение 2: Определите ошибку

Для каждого сценария определите, на какой стадии пайплайна произойдёт ошибка и что команда должна делать:

1. Разработчик запушил код с синтаксической ошибкой
2. Код разработчика компилируется, но ломает существующий unit-тест
3. Код проходит все тесты, но имеет уязвимость безопасности
4. Деплой в staging падает из-за нехватки места на диске
5. Всё проходит, но QA находит баг при ручном тестировании на staging

<details>
<summary>Нажмите, чтобы увидеть ответы</summary>

1. **Стадия сборки** — код не скомпилируется. Разработчик должен исправить синтаксическую ошибку.
2. **Стадия тестов** — unit-тесты падают. Разработчик должен исправить код или обновить тест, если изменение было намеренным.
3. **Стадия тестов** (проверка безопасности) — пайплайн должен упасть. Разработчик должен исправить уязвимость или обновить зависимость.
4. **Стадия деплоя в staging** — проблема инфраструктуры. DevOps должен очистить место на диске или увеличить хранилище.
5. **Не обнаружено пайплайном** — QA отклоняет деплой. Разработчик исправляет баг, пушит новый код, пайплайн запускается заново.

</details>

## 1. Что такое деплой?

**Деплой (развёртывание)** — это процесс размещения собранного приложения на сервере или в окружении, чтобы оно стало доступно для использования.

```
Собранный артефакт           ДЕПЛОЙ            Работающее приложение
(JAR, Docker-образ) ──────────────────→  (доступно по URL)
                     Скопировать, настроить,
                     запустить, проверить
```

### Деплой vs Сборка:
| Понятие | Что делает |
|---------|-----------|
| **Сборка (Build)** | Преобразует исходный код → артефакт (JAR, Docker-образ) |
| **Деплой (Deploy)** | Берёт артефакт → устанавливает и запускает его на сервере |

## 2. Окружения деплоя

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│    DEV      │───→│   QA/TEST  │───→│  STAGING   │───→│ PRODUCTION │
│             │    │            │    │            │    │            │
│ Разработчики│    │ QA-команда │    │ Финальная  │    │ Реальные   │
│ тестируют   │    │ тестирует  │    │ проверка   │    │ пользова-  │
│ код         │    │ функции    │    │ перед прод │    │ тели       │
│             │    │            │    │            │    │            │
│ Нестабильно │    │ Может      │    │ Похоже на  │    │ Должно быть│
│             │    │ ломаться   │    │ прод       │    │ стабильным │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
```

| Окружение | Назначение | Стабильность | Данные |
|-----------|-----------|-------------|--------|
| **Dev** | Тестирование разработчиками | Низкая — часто ломается | Фейковые/тестовые данные |
| **QA / Test** | Тестирование функций QA-командой | Средняя | Тестовые данные |
| **Staging** | Предпродакшен-валидация | Высокая — копия прода | Анонимизированные данные прода |
| **Production** | Реальные пользователи | Должна быть стабильной | Реальные данные |

### Конфигурация окружений:

У каждого окружения свои настройки:

```yaml
# Dev
database.url: jdbc:postgresql://dev-db:5432/myapp
api.url: https://dev.example.com/api
logging.level: DEBUG

# Staging
database.url: jdbc:postgresql://staging-db:5432/myapp
api.url: https://staging.example.com/api
logging.level: INFO

# Production
database.url: jdbc:postgresql://prod-db:5432/myapp
api.url: https://api.example.com
logging.level: WARN
```

## 3. Стратегии деплоя

### Стратегия 1: Rolling Deployment (последовательный деплой)

Замена экземпляров по одному:

```
Момент 0:  [v1] [v1] [v1] [v1]     ← Все работают на v1
Момент 1:  [v2] [v1] [v1] [v1]     ← Первый экземпляр обновлён
Момент 2:  [v2] [v2] [v1] [v1]     ← Второй экземпляр обновлён
Момент 3:  [v2] [v2] [v2] [v1]     ← Третий экземпляр обновлён
Момент 4:  [v2] [v2] [v2] [v2]     ← Все работают на v2 ✅
```

✅ Нулевой простой | ⚠️ Смешанные версии во время раскатки

### Стратегия 2: Blue-Green Deployment (сине-зелёный деплой)

Два идентичных окружения, переключение трафика:

```
До:        Трафик ──→ [BLUE: v1]     [GREEN: простаивает]
Деплой:    Трафик ──→ [BLUE: v1]     [GREEN: v2] ← Деплой v2 на green
Переключ.: Трафик ──→ [GREEN: v2]    [BLUE: v1]  ← Переключение трафика
Проверка:  Трафик ──→ [GREEN: v2]    [BLUE: v1]  ← Blue — вариант отката
```

✅ Мгновенный откат | ⚠️ Требуется двойная инфраструктура

### Стратегия 3: Canary Deployment (канареечный деплой)

Сначала направить небольшой % трафика на новую версию:

```
Шаг 1:   [v1 ← 95% трафика]   [v2 ← 5% трафика]    ← Тест на малой аудитории
Шаг 2:   [v1 ← 75% трафика]   [v2 ← 25% трафика]   ← Увеличиваем, если OK
Шаг 3:   [v1 ← 50% трафика]   [v2 ← 50% трафика]   ← Половина на половину
Шаг 4:   [v2 ← 100% трафика]                         ← Полная раскатка ✅
```

✅ Низкий риск — ловит проблемы рано | ⚠️ Сложно настроить

## 4. Контейнеры и Kubernetes

### Docker-контейнеры:

**Контейнер** упаковывает приложение со всем необходимым для запуска:

```
┌─────────────────────┐
│      Контейнер       │
│  ┌───────────────┐  │
│  │  Приложение    │  │
│  │  (my-app.jar)  │  │
│  ├───────────────┤  │
│  │  Java Runtime  │  │
│  ├───────────────┤  │
│  │  Linux (Alpine)│  │
│  └───────────────┘  │
└─────────────────────┘
```

**Зачем контейнеры?**
- 📦 **Консистентность** — один и тот же контейнер работает везде (dev, staging, prod)
- 🚀 **Скорость** — запуск за секунды
- 🔒 **Изоляция** — не мешает другим приложениям
- 📏 **Масштабируемость** — легко запустить несколько копий

### Kubernetes (K8s):

**Kubernetes** управляет контейнерами в продакшене — масштабирование, восстановление и маршрутизация.

```
┌─────────────────────────────────────────┐
│           КЛАСТЕР KUBERNETES             │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │  Pod      │  │  Pod      │  │ Pod    │ │
│  │ [my-app]  │  │ [my-app]  │  │[my-app]│ │
│  └──────────┘  └──────────┘  └────────┘ │
│        ↑              ↑            ↑     │
│        └──────────────┼────────────┘     │
│                 Балансировщик             │
│                  нагрузки                │
│                       ↑                  │
└───────────────────────┼──────────────────┘
                        │
                 Трафик пользователей
```

Ключевые понятия:
| Понятие | Что это |
|---------|--------|
| **Pod** | Минимальная единица — один или несколько контейнеров |
| **Deployment** | Управляет подами (масштабирование, обновления) |
| **Service** | Сетевая точка доступа к подам |
| **Namespace** | Логическая изоляция (dev, staging, prod) |

### Полезные команды kubectl для QA:

```bash
# Посмотреть, что запущено
kubectl get pods                        # Список всех подов
kubectl get pods -n staging             # Поды в пространстве имён staging

# Проверить состояние пода
kubectl describe pod my-app-abc123      # Подробная информация о поде
kubectl logs my-app-abc123              # Логи приложения
kubectl logs my-app-abc123 --tail=100   # Последние 100 строк

# Проверить деплойменты
kubectl get deployments                 # Список деплойментов
kubectl rollout status deployment/my-app # Проверить прогресс деплоя
```

## 5. Откаты (Rollback)

Когда деплой вызывает проблемы, выполняется **откат** к предыдущей версии.

### Когда откатывать:
- ❌ Приложение падает после деплоя
- ❌ Резкий рост количества ошибок
- ❌ Критическая функциональность сломана
- ❌ Значительное ухудшение производительности

### Как откатить:

```bash
# Откат в Kubernetes
kubectl rollout undo deployment/my-app

# Jenkins — перезапустить предыдущий успешный билд

# Docker — запустить контейнер с предыдущим тегом
docker run my-application:1.2.2    # Предыдущая версия
```

### Чек-лист отката для QA:

1. ✅ Убедиться, что откат завершён (предыдущая версия запущена)
2. ✅ Запустить смоук-тесты на откаченной версии
3. ✅ Убедиться, что исходная проблема исчезла
4. ✅ Сообщить, что произошло и что было замечено
5. ✅ Сохранить доказательства (скриншоты, логи, сообщения об ошибках)

## 6. Проверки работоспособности и мониторинг

После деплоя система должна убедиться, что приложение работает корректно.

### Эндпоинты проверки работоспособности:

```
GET /actuator/health        → { "status": "UP" }
GET /actuator/health/db     → { "status": "UP", "database": "PostgreSQL" }
GET /actuator/health/ready  → { "status": "UP" }  (готово принимать трафик)
GET /actuator/health/live   → { "status": "UP" }  (приложение живо)
```

### Что мониторить после деплоя:

| Метрика | На что смотреть |
|---------|----------------|
| **Частота ошибок** | Не должна увеличиваться после деплоя |
| **Время отклика** | Должно оставаться похожим или улучшиться |
| **CPU/Память** | Не должны неожиданно скачкообразно расти |
| **Проверки здоровья** | Должны возвращать "UP" |
| **Жалобы пользователей** | Мониторить каналы поддержки |
